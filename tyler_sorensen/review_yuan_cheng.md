# Review by Yuan Cheng (probabilistic_modeling_researcher)

## Project: LITMUS∞: Cross-Architecture Memory Model Portability Checker

**Reviewer Expertise:** Probabilistic modeling, statistical methodology, experimental design. Focus: statistical claims, confidence intervals, benchmark methodology.

**Overall Score:** weak accept

---

## Summary

LITMUS∞ commendably uses Wilson CIs for binomial proportions and bootstrap CIs for fence cost savings — genuine statistical literacy rare in systems verification. However, the experimental design has structural flaws that undermine the headline claims.

## Strengths

1. **Correct interval estimators.** Wilson intervals for boundary proportions (228/228, 50/50) avoid the known poor coverage of Wald intervals. Bootstrap CIs with BCa correction for non-binomial metrics correctly avoid normality assumptions.

2. **Multi-layered independence.** Five validation layers (enumeration, differential tests, herd7, SMT, Z3 certificates) provide genuinely independent evidence streams. A systematic error would need to evade all five simultaneously.

3. **Transparent disclosures.** The grounding.json honestly notes herd7 validation uses expected values not live execution, fence costs are analytical weights, and n=96 yields ±8% not ±5%. This epistemic honesty is uncommon.

## Weaknesses

1. **Fatal stratification problem in the 96-snippet benchmark.** The aggregate 81.2% exact-match hides catastrophic per-category failure: C++ atomics achieves **9% exact-match** (1/11). The Wilson CI [72.3%, 87.8%] treats snippets as IID from a homogeneous population, but per-category accuracy ranges from 9% to 100%. The correct approach is stratified estimation with between-category variance. A developer working primarily with `std::atomic` would experience 9% accuracy, not 81%. This 9% result should be in the abstract, not buried in a table.

2. **"Zero false negatives" is tautological.** The false-negative classification compares portability profiles of predicted vs. expected patterns — but both profiles are generated by the tool's own models. A predicted pattern could match the expected pattern's profile within the model while both being wrong about hardware. The claim is: "the AST analyzer introduces no additional false negatives beyond the model's existing ones." This is much weaker than "zero false negatives" implies.

3. **Fence savings variance invalidates mean-based claims.** ARM savings of 49% ± 33.6% yield a CV of 69%. The median (62.5%) differs substantially from the mean (49%), confirming skewness. The bootstrap CI [35.1%, 61.8%] spans 26.7pp. A developer could plausibly experience 0% to 90% savings. Without quantiles (Q25, median, Q75) or empirical CDFs, the mean is actively misleading.

4. **Timing conflates Python overhead with algorithm.** The 111ms for 570 pairs includes interpreter startup, imports, and data structure initialization. The "sub-1ms per pattern" claim requires per-query warm-start measurement, not batch time ÷ count. No cross-platform data is provided despite the tool targeting diverse hardware.

5. **Post-hoc power analysis is epistemologically vacuous.** Post-hoc power is a deterministic function of the CI and adds zero information (Hoenig & Heisey, 2001). The ">99% power" claim would hold for any observed result that happens to be significant. Prospective power analysis specifying the minimum detectable effect size is needed instead.

## Path to Best Paper

Three essential changes: (1) Report stratified per-category CIs with the worst-case category (9% C++ atomics) prominent. (2) Define false negatives relative to hardware ground truth or herd7, not relative to the tool's own models. (3) Report fence savings as empirical CDFs with quantiles, not means ± SDs. The statistical machinery is solid; the experimental design feeding into it needs fundamental restructuring.
