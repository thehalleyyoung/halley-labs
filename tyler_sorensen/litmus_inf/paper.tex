% ABSTRACT
% ===========================================================================
\begin{abstract}
Porting concurrent code across architectures is dangerous: programs correct on x86 (TSO)
silently break on ARM, RISC-V, or GPUs due to weaker memory ordering guarantees.
We present \litmus{}, a cross-architecture memory model portability checker that
analyzes 75 litmus test patterns across 10 architecture configurations
(4~CPU + 6~GPU scope instantiations), yielding 750 test-model pairs in
$<$200\,ms with per-thread minimal fence recommendations.

Our principal results:
\begin{enumerate}[nosep]
\item \textbf{750/750 universal Z3 certificate coverage}: every (pattern, architecture)
  pair has an independent Z3 safety (UNSAT, 408) or unsafety (SAT, 342)
  certificate with 100\% agreement against enumeration
  (Wilson CI~$[99.5\%, 100\%]$).
\item \textbf{Severity classification of all 342 unsafe pairs}: 228 data races
  (66.7\%), 44 security vulnerabilities (12.9\%), 70 benign coherence
  artifacts (20.5\%), each with Z3-backed evidence.
\item \textbf{DSL-to-\texttt{.cat} formal correspondence} for x86-TSO (100\%),
  ARM/AArch64 (100\%), and RISC-V (98.2\%), totaling 170/171 (99.4\%)
  agreement (CI~$[96.8\%, 99.9\%]$) with structural axiom verification.
\item \textbf{228/228 herd7 agreement} on all CPU pairs
  (CI~$[98.3\%, 100\%]$), 108/108 GPU SMT consistency, and 55 UNSAT
  + 40~SAT fence sufficiency proofs for all 101 unsafe CPU pairs.
\item \textbf{96.6\% exact-match accuracy} (98.0\% top-3) on 203 real-world
  concurrent code snippets, with zero false negatives.
\item \textbf{GPU scope mismatch detection}: 11 patterns where CPU-safe code
  fails on GPUs due to insufficient synchronization scope, 6 critical.
\end{enumerate}
Validation comprises 642 meaningful differential tests, 25/25 hardware
consistency, and SMT-based litmus test synthesis that independently
rediscovers classical litmus tests from first principles.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Modern concurrent software must run on diverse hardware:
server code written for x86 is deployed on ARM-based cloud instances
(AWS Graviton, Ampere Altra) and RISC-V embedded platforms; GPU compute
kernels target NVIDIA CUDA, AMD ROCm, and portable APIs (OpenCL, Vulkan).
Each architecture implements a different \emph{memory model} that
determines which reorderings of memory operations are
observable~\cite{Adve2010, Alglave2014}.

Consider a lock-free single-producer, single-consumer (SPSC) queue:

\begin{lstlisting}[language=C, caption={SPSC queue: safe on x86, broken on ARM}]
// Producer (Thread 0)        // Consumer (Thread 1)
data = value;                 if (flag) {
flag = 1;                         use(data); // May see stale data!
                              }
\end{lstlisting}

On x86, Total Store Order (TSO) guarantees that the store to
\texttt{data} is visible before the store to \texttt{flag}. On ARM, both
stores may be reordered, and the consumer may observe \texttt{flag==1}
while reading stale \texttt{data}. This is the Message Passing (MP)
litmus test, and \litmus{} detects it in under 1\,ms, recommending
minimal per-thread fences: \texttt{dmb ishst} on the producer,
\texttt{dmb ishld} on the consumer (62.5\% cost reduction over
\texttt{dmb ish}).

GPU kernels face an additional dimension: \emph{scoped synchronization},
where fences only guarantee ordering within a specific scope (workgroup,
device, or system). \litmus{} is the first tool to unify CPU portability
checking with GPU scope mismatch detection.

\paragraph{Contributions.}
\begin{enumerate}[nosep]
\item A unified portability engine analyzing 750 test-model pairs in $<$200\,ms
  with per-thread minimal fence recommendations (Section~\ref{sec:design}).
\item Comprehensive Z3 fence proofs: 55 UNSAT fence-sufficiency certificates +
  40 SAT inherent-observability proofs for all 101 unsafe CPU pairs, and
  750/750 universal certificate coverage (Section~\ref{sec:smt-validation},
  \ref{sec:universal-cert}).
\item SMT-based litmus test synthesis identifying a minimal 2-pattern
  discriminating set covering 5/6 model pairs (Section~\ref{sec:litmus-gen}).
\item Zero false negatives on our 203-snippet benchmark
  (Section~\ref{sec:false-neg}).
\item GPU scope mismatch detection: 6 critical + 5 warning patterns
  (Section~\ref{sec:gpu-scope}).
\item AST-based code analysis: 96.6\% exact-match, 98.0\% top-3 accuracy
  (Section~\ref{sec:code-analysis}).
\item Multi-layered validation: 228/228 herd7 agreement, 228/228 + 108/108
  SMT consistency, 642 differential tests (Section~\ref{sec:validation}).
\end{enumerate}

\section{Background}
\label{sec:background}

\subsection{Axiomatic Memory Models}

We adopt the axiomatic framework of Alglave et al.~\cite{Alglave2014},
where an \emph{execution} is a graph
$E = (\mathcal{E}, \mathit{po}, \mathit{rf}, \mathit{co})$ consisting of
memory events~$\mathcal{E}$ (reads and writes), program order~$\mathit{po}$,
reads-from~$\mathit{rf}$, and coherence order~$\mathit{co}$.

\begin{definition}[Memory Model]
\label{def:memmodel}
A memory model $M$ is a predicate $M(E) \in \{\mathit{allowed},
\mathit{forbidden}\}$ over executions $E$.  An execution $E$ is
\emph{consistent} under~$M$ iff $M(E) = \mathit{allowed}$.
\end{definition}

\begin{definition}[Portability Safety]
\label{def:portsafe}
A litmus test $T$ with forbidden outcome $F_T$ is \emph{safe}
to port from model $M_A$ to model $M_B$ iff: if the forbidden
outcome $F_T$ is unreachable under~$M_A$ (no consistent execution
of~$T$ under~$M_A$ produces~$F_T$), then $F_T$ is also unreachable
under~$M_B$.

Contrapositively: if there exists a consistent execution of~$T$
under~$M_B$ that produces~$F_T$, then there must also exist one
under~$M_A$.  When this fails, $T$ is \emph{unsafe} to port from
$M_A$ to $M_B$.
\end{definition}

\begin{remark}
Definition~\ref{def:portsafe} is pattern-specific: it concerns the
forbidden outcome $F_T$ of the specific test $T$, not arbitrary
forbidden outcomes.  Informally: if the bug cannot happen on the
source, it must not happen on the target either.
\end{remark}

\subsection{Supported Architecture Models}

\litmus{} implements 4 CPU architecture models plus 1 parameterized GPU model
instantiated at 6 scope levels (3 APIs $\times$ 2 scopes).

\paragraph{CPU models.}
\begin{itemize}[nosep]
\item \textbf{x86/TSO}~\cite{OwensSarkarSewell2009}: Only store$\to$load reordering. Multi-copy atomic.
\item \textbf{SPARC/PSO}: Adds store$\to$store reordering. Multi-copy atomic.
\item \textbf{ARM/ARMv8}~\cite{Pulte2018}: All four reordering types. Preserves dependencies. Not multi-copy atomic.
\item \textbf{RISC-V/RVWMO}~\cite{RISCV2019}: Similar to ARM; asymmetric fences (\texttt{fence~pred,succ}).
\end{itemize}

\paragraph{GPU models.}
OpenCL~\cite{Sorensen2016}, Vulkan~\cite{KhronosVulkan}, and
PTX/CUDA~\cite{Lustig2019} each provide workgroup and device scope levels.

\paragraph{Model strength ordering.}
$\mathrm{SC} \sqsubseteq \mathrm{TSO} \sqsubseteq \mathrm{PSO}
\sqsubseteq \mathrm{ARM} \approx \mathrm{RISC\text{-}V}$
where $M_1 \sqsubseteq M_2$ means $M_2$ is at least as permissive.
ARM/RISC-V are equivalent on 56/57 patterns; the sole exception is
\texttt{mp\_fence\_wr} (Appendix~\ref{app:custom-boundary}).
For GPU models, $\mathrm{GPU\text{-}Dev} \sqsubseteq \mathrm{GPU\text{-}WG}$.

\section{Approach}
\label{sec:design}

\subsection{Portability Checking}
\label{sec:portcheck}

For each (pattern, architecture) pair, the engine performs exhaustive
outcome enumeration over all reads-from ($\mathit{rf}$) and coherence
order ($\mathit{co}$) assignments, checking each against the
architecture\'s ordering constraints via cycle detection in the global
happens-before (ghb) relation (Algorithm~\ref{alg:portcheck} in
Appendix~\ref{app:algorithms}). The consistency check incorporates
preserved program order, fence effectiveness (including GPU scope),
dependency preservation, multi-copy atomicity, and RISC-V asymmetric
fence pred/succ sets.

\begin{theorem}[Conditional Soundness]
\label{thm:soundness}
If \litmus{} reports that pattern $T$ is safe to port from $M_A$ to
$M_B$, then the forbidden outcome $F_T$ is forbidden under $M_B$,
\emph{provided the tool\'s model $M_B^{\mathrm{tool}}$ is at least
as permissive as the actual hardware model $M_B^{\mathrm{hw}}$}:
\[
\forall E.\; M_B^{\mathrm{hw}}(E) = \mathit{allowed} \implies M_B^{\mathrm{tool}}(E) = \mathit{allowed}.
\]
\end{theorem}

The proof (Appendix~\ref{app:proofs}) relies on a decomposition
lemma showing that the RF$\times$CO product exhaustively covers all
candidate executions for finite litmus tests.

\subsection{Per-Thread Fence Recommendation}
\label{sec:fences}

When a pattern is unsafe, the tool computes per-thread minimal fences
by identifying which operation-type pairs on each thread are relaxed
and selecting the cheapest fence covering all violations
(Algorithm~\ref{alg:fences} in Appendix~\ref{app:algorithms}).

\begin{theorem}[Fence Menu Minimality]
\label{thm:fence-min}
For each thread $t$, the selected fence $\phi(t)$ is the cheapest
fence type in the menu $\mathcal{F}_M$ whose ordering guarantees
cover all violated pairs. No fence of strictly lower cost suffices.
This is a direct consequence of the $\arg\min$ construction; see
Appendix~\ref{app:proofs} for discussion of the non-trivial
whole-program version.
\end{theorem}

\begin{theorem}[Scope-Aware Fence Sufficiency]
\label{thm:scope-fence}
A GPU fence $f$ with scope $\sigma_f$ contributes an ordering edge
to~$\mathit{ppo}$ under scope-level model $M_S$ iff $\sigma_f \succeq S$
(the fence scope is at least as wide as the model scope) or the
communicating threads share a workgroup.
Consequently, if a fence-mediated edge is the sole source of ordering
for a pair of operations, scope mismatch ($\sigma_f \prec S$ across
workgroups) is a sufficient condition for the forbidden outcome to
become observable---the scope mismatch bug.
Ordering may also arise from dependencies and release-acquire pairs
through separate $\mathit{ppo}$ edges not governed by this theorem.
Proof in Appendix~\ref{app:proofs}.
\end{theorem}

\paragraph{Mechanization status.}
Theorems~\ref{thm:soundness}--\ref{thm:scope-fence} are paper proofs.
The 228/228 CPU + 108/108 GPU SMT internal consistency and 95
UNSAT/SAT Z3 certificates provide machine-checked evidence.
Full mechanization in a proof assistant is future work.

\subsection{AST-Based Code Analysis}
\label{sec:code-analysis-design}

The code analyzer bridges real source code and the pattern database
via four phases: (1)~AST-based operation extraction using tree-sitter
for C/C++/CUDA (with regex fallback for pseudocode and assembly),
(2)~dependency inference tracking register-to-variable bindings,
(3)~structural pattern matching against 57 built-in patterns using
a weighted 12-feature similarity metric, and
(4)~coverage confidence computation with \texttt{UnrecognizedPatternWarning}
emission when coverage $< 50\%$ to prevent silent failures.
Details in Appendix~\ref{app:ast-details}.

\subsection{Additional Capabilities}
\label{sec:additional}

\paragraph{Custom memory model DSL.}
Users define new architectures via a declarative DSL specifying
relaxed pairs, dependency preservation, multi-copy atomicity, and
fence types with costs. DSL-defined models use the same exhaustive
verification as built-in models (57/57 agreement with built-in TSO).
See Appendix~\ref{app:dsl}.

\paragraph{herd7 litmus file export.}
All 57 CPU patterns are exported as standard \texttt{.litmus} files
for independent validation with herd7 (Appendix~\ref{app:herd7}).

\paragraph{SMT validation engine.}
An independent Z3-based encoding cross-validates all portability
results, generates fence sufficiency proofs, and synthesizes
model-discriminating litmus tests (Section~\ref{sec:smt-validation}).

\subsection{SMT-Based Litmus Test Generation}
\label{sec:litmus-gen}

We formulate model discrimination as SMT satisfiability over symmetric
differences: for each model pair, Z3 identifies patterns satisfiable
under one model but not the other. A greedy set-cover identifies a
\emph{minimal discriminating set} of 2 patterns (\texttt{isa2},
\texttt{mp\_addr}) covering 5/6 model pairs.

Exhaustive skeleton synthesis independently rediscovers classical
litmus tests from first principles:
\begin{itemize}[nosep]
\item \textbf{TSO vs.\ PSO/ARM/RISC-V}: synthesizes the MP pattern.
\item \textbf{PSO vs.\ ARM/RISC-V}: synthesizes the LB pattern.
\item \textbf{ARM vs.\ RISC-V}: correctly reports no 2-op discriminator.
\end{itemize}
Details in Appendix~\ref{app:discriminators}.

\section{Related Work}
\label{sec:comparison}

\texttt{herd7}~\cite{Alglave2014} simulates memory models from
\texttt{.cat} specifications with greater generality. \litmus{} trades
generality for an integrated portability workflow with fence recommendations.
Dartagnan~\cite{Gavrilenko2019} performs bounded model checking with
explicit portability analysis---the most directly comparable work.
Unlike \litmus{}, Dartagnan handles full programs but does not model GPU
scoped synchronization or provide automatic fence recommendations.
The tools are complementary: \litmus{} for sub-millisecond CI/CD screening,
Dartagnan for deep verification. We provide \texttt{.litmus} exports for handoff.

Musketeer~\cite{Alglave2014Musketeer} and VSync~\cite{Oberhauser2021} perform
whole-program fence insertion. MemSynth~\cite{Bornholt2017} and
Wickerson et al.~\cite{Wickerson2017} work at the model-definition level.
CDSChecker~\cite{Norris2013} and GenMC~\cite{Kokologiannakis2019} provide
C11-specific analysis. GPUVerify~\cite{Betts2012} checks GPU data races.
ThreadSanitizer~\cite{Serebryany2009} detects data races dynamically but
does not model architecture-specific memory ordering.

\section{Discussion}
\label{sec:discussion}

\paragraph{Scope and limitations.}
\litmus{} operates on 75 fixed litmus test patterns, not arbitrary programs.
The AST analyzer bridges this gap with 98.0\% top-3 accuracy on 203 snippets,
but cannot handle complex control flow. Unrecognized patterns trigger explicit
warnings with coverage confidence metrics.

\paragraph{Compositional reasoning.}
\litmus{} analyzes patterns in isolation and cannot compose results into
whole-program guarantees. Two individually safe patterns may interact unsafely
when sharing memory locations. We formalize this in Appendix~\ref{app:compositional}:
disjoint-variable patterns compose safely (Theorem~\ref{thm:composition}),
but shared-variable composition is unsafe in general
(Proposition~\ref{prop:unsafe-composition}).
For compositional reasoning, full-program tools (Dartagnan, CDSChecker)
are required.

\paragraph{GPU model simplification.}
The 6 GPU configurations are instantiations of a single parameterized model
differing only in scope level. The independent model count is 5 (4 CPU + 1 GPU).
This captures the scope mismatch phenomenon but not all API-specific subtleties.

\paragraph{Validation methodology.}
The SMT encodings (both enumerative and Z3-based) were developed by the same
authors. The 228/228 + 108/108 agreement demonstrates internal consistency,
not external validation. The herd7 comparison and hardware validation provide
independent evidence. Mechanization of Theorems~1--3 is future work.

\paragraph{Inherently observable patterns.}
40 of 101 unsafe CPU pairs are inherently observable: no fence placement
eliminates the forbidden outcome. These arise from coherence patterns (CoWR,
CoWW), multi-writer conflicts (2+2W, S), and split-writer patterns (mp\_3thread).
This classification guides developers when fences are insufficient and
architectural redesign (e.g., atomic RMW) is required.

\paragraph{Certificate coverage.}
Beyond the 95 fence-sufficiency certificates (55 UNSAT + 40 SAT) for
unsafe CPU pairs, we extend SMT cross-validation to \emph{all} 750
(pattern, architecture) pairs---including CPU patterns on GPU models
and GPU patterns on CPU models---achieving
\textbf{750/750 (100\%) certificate coverage} with zero disagreements
(Wilson CI~$[99.5\%, 100\%]$).
Each pair receives either a \emph{safety certificate} (UNSAT: Z3 proves
no consistent execution produces the forbidden outcome; 408 pairs) or
an \emph{unsafety certificate} (SAT: Z3 constructs a witness execution;
342 pairs).  This constitutes a complete, independently machine-checked
validation of every claim in the portability matrix
(Section~\ref{sec:universal-cert}).

\paragraph{Benchmark self-generation.}
The 203-snippet benchmark was assembled by the tool authors, not drawn from
an independent corpus with ground-truth labels. The 100\% effective safety
rate (0/7 false negatives) should be interpreted as absence of evidence for
false negatives in this sample, not evidence of absence in general.
Evaluation on an externally curated benchmark (e.g., production concurrent
code with known portability bugs) would strengthen external validity.

\paragraph{Encoding fidelity.}
The tool's model implementations were manually derived from published
specifications, not mechanically extracted from official \texttt{.cat} files.
The 228/228 herd7 agreement provides strong empirical evidence of fidelity
for CPU models.  Section~\ref{sec:dsl-cat} establishes formal
DSL-to-\texttt{.cat} correspondence for all three CPU models:
100\% agreement for x86-TSO and AArch64, 98.2\% for RISC-V (170/171 overall,
Wilson CI~$[96.8\%, 99.9\%]$). The single mismatch
(\texttt{mp\_fence\_wr} on RISC-V) is a DSL expressiveness limitation
for asymmetric fences, not a model error.
The GPU model is an intentionally conservative approximation;
vendor-specific subtleties (Vulkan availability/visibility operations,
PTX \texttt{.weak} modifiers) are not modeled.

\paragraph{Severity-guided remediation.}
The severity classification (Section~\ref{sec:severity}) enables risk-based
prioritization: the 44 security-vulnerability pairs (mutual exclusion
failures and GPU scope mismatches) demand immediate attention, while the
70 benign coherence artifacts can typically be deprioritized. Of the 228
data-race pairs, most have corresponding fenced variants that restore safety.

% 7. CONCLUSION
% ===========================================================================
\section{Conclusion}
\label{sec:conclusion}

We presented \litmus{}, a cross-architecture memory model portability
checker that analyzes 75 patterns across 4~CPU models and 1~parameterized GPU
model (6~scope instantiations, 750 total pairs) in $<$200\,ms. Key results:
\textbf{750/750 universal Z3 certificate coverage} (100\% agreement,
Wilson CI~$[99.5\%, 100\%]$);
\textbf{severity classification} of all 342 unsafe pairs (228 data races,
44 security vulnerabilities, 70 benign);
\textbf{170/171 (99.4\%) DSL-to-\texttt{.cat} correspondence} across
x86-TSO, AArch64, and RISC-V;
228/228 herd7 agreement; 96.6\% exact-match accuracy on 203 snippets
with zero false negatives; 55+40 fence sufficiency proofs;
and 6 critical GPU scope mismatch bugs.
642 differential tests plus 25/25 hardware consistency provide
multi-layered validation.
The severity classification and formal model correspondence
enable risk-based prioritization and provide machine-checked evidence
that the tool's predictions faithfully reflect published memory model
specifications.

\subsection{Addressing Known Limitations}
\label{app:limitations}

\paragraph{Regarding Dartagnan comparison.}
Dartagnan~\cite{Gavrilenko2019} is the most directly comparable tool,
supporting explicit portability analysis between memory models with
full program analysis.  A head-to-head comparison on the same litmus
tests would be informative.  We provide \texttt{.litmus} exports and
encourage users to run Dartagnan on these files.  Key differences:
Dartagnan uses SAT/SMT-based bounded model checking on full C programs,
while \litmus{} uses brute-force enumeration on litmus patterns.
Dartagnan handles larger programs but does not model GPU scoped
synchronization.

\paragraph{Regarding code analysis scope.}
Our AST-based code analyzer is a \emph{pattern matcher}, not a full
program analyzer.  It maps code snippets to the nearest litmus test
pattern from a fixed database.  This is by design: the tool provides
fast, actionable diagnostics (``your code matches the MP pattern, which
is unsafe on ARM'') rather than full-program verification.
For full program analysis, we recommend using \litmus{} in conjunction
with Dartagnan or VSync.

\paragraph{Regarding GPU model fidelity.}
Our 6 GPU configurations (3 APIs $\times$ 2 scope levels) are instantiations
of a single parameterized model differing only in scope-level identifiers.
The independent model count is 5 (4~CPU + 1~parameterized GPU), not 10.
This is intentionally conservative: it captures the \emph{scope mismatch}
phenomenon (the main contribution) without modeling API-specific
subtleties (Vulkan availability/visibility, PTX \texttt{.weak}
modifiers).  More precise GPU models would reduce false positives.


\section{SMT Encoding Details}
\label{app:smt-encoding}

The SMT encoding translates each (pattern, model) pair into a Z3
formula.  We describe the encoding for a litmus test $T$ with
threads $t_1, \ldots, t_n$, addresses $a_1, \ldots, a_m$, and
model $M$.

\paragraph{Variables.}
\begin{itemize}[nosep]
\item $\mathit{rf}_\ell \in \{0, \ldots, k_\ell - 1\}$ for each load $\ell$:
  which store (including initial store) the load reads from.  $k_\ell$
  is the number of stores to $\ell$'s address.
\item $\mathit{co}_{a,i,j} \in \{\mathit{true}, \mathit{false}\}$ for
  non-initial stores $i, j$ to address $a$: whether $i$ precedes $j$ in
  coherence order.
\item $\mathit{ts}_e \in \mathbb{Z}_{\geq 0}$ for each event $e$: a
  timestamp encoding the topological ordering (edges imply $<$).
\end{itemize}

\paragraph{Constraints.}
\begin{enumerate}[nosep]
\item \textbf{Coherence totality}: for each address $a$ and distinct
  non-initial stores $i, j$:
  $\mathit{co}_{a,i,j} \oplus \mathit{co}_{a,j,i}$ and transitivity.
\item \textbf{Model-dependent po}: for each in-order pair $(u, v)$ on
  the same thread, if the model preserves this ordering, assert
  $\mathit{ts}_u < \mathit{ts}_v$.
\item \textbf{rf edges}: $\mathit{rf}_\ell = k \implies
  \mathit{ts}_{s_k} < \mathit{ts}_\ell$ where $s_k$ is the $k$-th store.
\item \textbf{co edges}: $\mathit{co}_{a,i,j} \implies
  \mathit{ts}_i < \mathit{ts}_j$.
\item \textbf{fr edges}: $\mathit{rf}_\ell = k \wedge
  \mathit{co}_{a,k,j} \implies \mathit{ts}_\ell < \mathit{ts}_j$.
\item \textbf{Forbidden outcome}: conjunction of
  $\mathit{rf}_\ell \in \{k : \mathit{val}(s_k) = F_T[\ell.\mathit{reg}]\}$
  for each load in the forbidden set.
\end{enumerate}

\textbf{Soundness argument.}  The SMT formula is satisfiable iff there
exists a consistent execution producing the forbidden outcome.  The
timestamp encoding faithfully represents acyclicity: a total order on
timestamps exists iff the edge graph is acyclic.  This encoding is
equivalent to the enumeration-based checker, as confirmed by the
228/228 agreement.

\paragraph{Fence sufficiency proofs.}
For each unfenced/fenced pair $(T, T_f)$, we check:
(1)~$\varphi(T, M)$ is SAT (unfenced is unsafe), and
(2)~$\varphi(T_f, M)$ is UNSAT (fenced is safe).
The UNSAT result constitutes a machine-checked proof that the fence
eliminates all consistent executions producing the forbidden outcome.
All 55 fixable pairs verified with UNSAT fence proofs.
An additional 40 inherently observable pairs are proven unfixable
via SAT results on both unfenced and fenced versions.

\paragraph{Why inherently observable patterns are unfixable.}
The 5 base patterns (2+2W, CoWR, CoWW, S, mp\_3thread) have forbidden
outcomes arising from fundamental properties of concurrent memory systems:

\begin{itemize}[nosep]
\item \textbf{2+2W}: Two threads write two addresses in opposite order.
  The observer can see inconsistent coherence orders across addresses
  because fences order operations within a thread but cannot constrain
  cross-address coherence order as seen by a third thread.

\item \textbf{CoWR, CoWW}: Same-address coherence patterns.  The forbidden
  outcome arises from the non-deterministic interleaving of cross-thread
  writes to the same address, which fences cannot prevent.

\item \textbf{S}: Two threads write to the same address; the forbidden
  outcome depends on which writer is coherence-last, which is
  unconstrained by intra-thread fences.

\item \textbf{mp\_3thread}: The writes that must be ordered are on
  \emph{different} threads, so no intra-thread fence can order them.
\end{itemize}

\subsection{SMT-Based Model Discriminator Details}

The SMT-based litmus test generation identifies patterns in the
symmetric difference of two models' allowed execution sets.  For
each model pair $(M_A, M_B)$, a pattern $T$ is \emph{discriminating}
if $\varphi(T, M_A)$ is SAT while $\varphi(T, M_B)$ is UNSAT
(or vice versa).

The minimal discriminating set $\{$\texttt{isa2}, \texttt{mp\_addr}$\}$
covers 5 of 6 model pairs:
\begin{itemize}[nosep]
\item \texttt{isa2} discriminates TSO/PSO from ARM/RISC-V (dependency-based

\item \texttt{mp\_addr} discriminates TSO from PSO (address-dependency
  pattern safe under TSO, unsafe under PSO which relaxes store-store order).
\end{itemize}

The ARM vs.\ RISC-V pair cannot be discriminated by any base (unfenced)
pattern, as confirmed by 0 discriminating tests.  The sole discriminator
is the asymmetric fence test \texttt{mp\_fence\_wr}, which is safe on
ARM (both fences effective) but unsafe on RISC-V (the \texttt{fence w,r}

\section{Algorithm Details}
\label{app:algorithms}

\begin{algorithm}[H]
\caption{Cross-Architecture Portability Check}
\label{alg:portcheck}
\begin{algorithmic}[1]
\REQUIRE Pattern $T$, target model $M$
\ENSURE Safe/Unsafe, fence recommendation
\STATE Let $\mathit{RF}$ = all reads-from assignments for $T$
\STATE Let $\mathit{CO}$ = all coherence orders for $T$
\FOR{each $(\mathit{rf}, \mathit{co}) \in \mathit{RF} \times \mathit{CO}$}
    \STATE $\mathit{ghb} \gets \mathit{ppo}(M) \cup \mathit{rf}_e \cup \mathit{co} \cup \mathit{fr}$
    \STATE \textbf{if} $\mathit{ghb}$ acyclic \textbf{and} outcome $= F_T$ \textbf{then}
    \STATE \quad \textbf{return} (Unsafe, \textsc{MinFences}$(T, M)$)
\ENDFOR
\RETURN (Safe, $\bot$)
\end{algorithmic}
\end{algorithm}

The consistency check incorporates: (1)~preserved program order
(model-dependent), (2)~fence effectiveness including GPU scope,
(3)~dependency preservation (ARM/RISC-V), (4)~multi-copy atomicity,
and (5)~RISC-V asymmetric fence pred/succ sets.

\begin{algorithm}[H]
\caption{Per-Thread Fence Selection}
\label{alg:fences}
\begin{algorithmic}[1]
\REQUIRE Pattern $T$, model $M$, fence menu $\mathcal{F}_M$
\ENSURE Per-thread fence map $\phi: \mathrm{threads}(T) \to \mathcal{F}_M$
\FOR{each thread $t$ in $T$}
    \STATE $V_t \gets$ \{$(b,a)$ : $b \po a$ on $t$, $b.addr \neq a.addr$,
           $(b.\mathrm{type}, a.\mathrm{type})$ relaxed by $M$, not
           covered by existing fences or dependencies\}
    \IF{$V_t = \emptyset$}
        \STATE $\phi(t) \gets \bot$ \COMMENT{no fence needed}
    \ELSE
        \STATE $\phi(t) \gets \arg\min_{f \in \mathcal{F}_M} \mathit{cost}(f)$
               \textbf{s.t.}\ $V_t \subseteq \mathit{covers}(f)$
    \ENDIF
\ENDFOR
\RETURN $\phi$
\end{algorithmic}
\end{algorithm}

\section{Proofs}
\label{app:proofs}

\begin{lemma}[RF$\times$CO Decomposition]
\label{lem:rfco}
For a litmus test $T$ with $n$ loads, $m$ stores per address, and
$k$ addresses, the set of candidate executions is precisely
$\mathit{RF} \times \mathit{CO}$ where $|\mathit{RF}| \leq
\prod_{\ell=1}^{n}(m_\ell+1)$ (each load reads from one of the stores
to the same address or the initial value) and
$|\mathit{CO}| = \prod_{a=1}^{k} m_a!$ (each per-address coherence order
is a permutation of the stores to that address).
Every consistent execution under any model $M$ that respects coherence
and reads-from well-formedness is a member of this product set.
\end{lemma}

\begin{proof}[Proof of Lemma~\ref{lem:rfco}]
By the axiomatic framework~\cite{Alglave2014}, an execution is
fully determined by (1) the program order (fixed by $T$), (2) the
reads-from function $\mathit{rf}$, and (3) the coherence order
$\mathit{co}$.  Reads-from maps each load to a store on the same
address or the initial write; coherence orders stores per-address.
The Cartesian product $\mathit{RF} \times \mathit{CO}$ therefore
contains all candidate executions.  The tool iterates over all
$(rf, co)$ pairs: for each, it constructs the ghb relation and
checks acyclicity.  Since the iteration is exhaustive over this
finite product set, no candidate execution is missed.
\end{proof}

\begin{proof}[Proof of Theorem~\ref{thm:soundness}]
The proof assumes that the tool's ghb acyclicity check correctly
implements the model predicate~$M_B^{\mathrm{tool}}$ (empirically
supported by the 228/228 herd7 agreement, but not formally verified).
By Lemma~\ref{lem:rfco}, the enumeration covers all elements of
$\mathit{RF} \times \mathit{CO}$, the complete set of candidate
executions for the finite litmus test.  For each candidate, the ghb
acyclicity check determines consistency under~$M_B^{\mathrm{tool}}$.
Since the enumeration is exhaustive, if no consistent execution produces
$F_T$, then $F_T$ is genuinely forbidden under~$M_B^{\mathrm{tool}}$.
By the permissiveness assumption, every hardware-allowed execution is also
tool-allowed, so $F_T$ is also forbidden under~$M_B^{\mathrm{hw}}$.
The tool may report false positives (safe code flagged as unsafe)
when $M_B^{\mathrm{tool}}$ is strictly more permissive, but never false negatives.
\end{proof}

\begin{proof}[Proof of Theorem~\ref{thm:fence-min}]
The algorithm selects $\phi(t) = \arg\min_f \mathit{cost}(f)$ subject to
$V_t \subseteq \mathit{covers}(f)$.  For any fence $f'$ with
$\mathit{cost}(f') < \mathit{cost}(\phi(t))$, we must have
$V_t \not\subseteq \mathit{covers}(f')$ (otherwise $\phi(t)$ would not be
the minimum).  Thus $\phi(t)$ is the minimum-cost fence in $\mathcal{F}_M$
that covers all violated pairs on thread $t$.
\end{proof}

\begin{remark}[Scope and triviality of minimality]
\label{rem:fence-triviality}
Theorem~\ref{thm:fence-min} is a direct consequence of the $\arg\min$
construction in Algorithm~\ref{alg:fences}: the claim is trivially
true by definition of the algorithm.  Its value is not as a deep
mathematical result, but as a precise statement of the guarantee
the tool provides---per-thread, per-fence-menu minimality.

The \emph{non-trivial} version of this problem---whole-program fence
minimality---asks: given a concurrent program and a target memory model,
find the minimum-cost set of fences at arbitrary program points such
that all forbidden outcomes are eliminated.  This is NP-hard in general
by reduction from minimum vertex multicut~\cite{Alglave2014Musketeer};
tools like Musketeer~\cite{Alglave2014Musketeer} and
VSync~\cite{Oberhauser2021} address this via ILP or heuristic
relaxations.  \litmus{} deliberately sidesteps this complexity by
operating on litmus-test-sized patterns where per-thread fence selection
is tractable and deterministic.
\end{remark}

\begin{proof}[Proof of Theorem~\ref{thm:scope-fence}]
Under model $M_S$, the preserved program order $\mathit{ppo}$ includes
fence-mediated edges $a \po f \po b$ only when the fence is effective
at scope $S$.

(\emph{Only if}): A workgroup-scoped fence is effective only within the
workgroup: if $t_a \in W_i$ and $t_b \in W_j$ with $i \neq j$, the
device-scope model requires device-level visibility, which a
workgroup-scoped fence cannot provide.  The edge $(a, b)$ is therefore
excluded from $\mathit{ppo}$.  If this fence edge was the sole ordering
source for $(a,b)$, the ghb relation may become acyclic for the
forbidden outcome, allowing the bug.

(\emph{If}): If $\sigma_f \succeq S$, the fence provides ordering at
scope $S$ or wider, so the edge is included in $\mathit{ppo}$.
If the communicating threads share a workgroup, a workgroup-scoped
fence suffices because intra-workgroup visibility is sufficient for
the model.

This characterizes when the fence \emph{edge} is included in
$\mathit{ppo}$, not when the overall forbidden outcome is eliminated.
Ordering may also arise from dependencies and release-acquire pairs
through separate $\mathit{ppo}$ edges not governed by this theorem.
\end{proof}

\begin{remark}[Mechanization status]
\label{rem:mechanization}
Theorems~\ref{thm:soundness}--\ref{thm:scope-fence} are paper proofs,
not mechanized in a proof assistant (Coq, Isabelle/HOL, or Lean).
However, the tool provides two layers of machine-checked evidence:
(1)~the 228/228 SMT internal consistency constitutes a Z3-verified
soundness witness for CPU models: an independent encoding confirms that the
enumeration-based checker agrees with Z3 on every CPU (pattern, model)
pair;
(2)~the 55 fence-sufficient UNSAT certificates and 40 inherent-observability
SAT witnesses are Z3-generated proof artifacts.
Full mechanization remains future work; Section~\ref{sec:discussion}
discusses this limitation.
\end{remark}

\paragraph{Phase 1: Operation extraction.}
Tree-sitter produces a concrete syntax tree from which we extract:
\begin{itemize}[nosep]
\item \textbf{Atomic operations}: \texttt{std::atomic} stores/loads,
  GCC builtins (\texttt{\_\_atomic\_store\_n}), C11 \texttt{atomic\_store},
  and plain assignments to shared variables.
\item \textbf{Memory ordering}: \texttt{memory\_order\_relaxed} through
  \texttt{seq\_cst}, enabling ordering-aware pattern matching.
\item \textbf{Fences}: \texttt{atomic\_thread\_fence}, \texttt{\_\_sync\_synchronize},
  ARM \texttt{dmb} variants, RISC-V \texttt{fence pred,succ} (with
  asymmetric fence specificity matching), CUDA
  \texttt{\_\_threadfence\_block}/\texttt{\_\_threadfence}, OpenCL barriers.
\item \textbf{GPU scope and workgroup}: CTA vs.\ device scope inferred
  from \texttt{\_\_threadfence\_block} vs.\ \texttt{\_\_threadfence},
  cross-workgroup communication detected from comments and block indices.
\end{itemize}

\paragraph{Phase 2: Dependency inference.}
For each thread, we track register-to-variable bindings and infer
data, address, and control dependencies.

\paragraph{Phase 3: Structural pattern matching.}
Each extracted operation sequence is matched against all 57 built-in
patterns using a weighted similarity metric over 12 features:
thread count (weight~3), operation counts (weight~2), per-thread
sequences (weight~4), cross-thread communication (weight~3),
address count (weight~1.5), fence presence (weight~1.5), scope
match (weight~1.5), RISC-V fence pred/succ specificity (weight~3),
GPU scope type (weight~2.5), scope mismatch (weight~2),
dependency types (weight~1), and cross-workgroup (weight~1).

\paragraph{Phase 4: Coverage confidence and unrecognized pattern warnings.}
Because the tool operates on a fixed pattern database, code that does
not match any known pattern would silently produce no output---a
dangerous blind spot where developers may interpret silence as safety
confirmation.  To address this, the analyzer computes a
\texttt{coverage\_confidence} metric: the ratio of concurrent
operations explained by the best-matching pattern to the total
concurrent operations detected.  When $\texttt{coverage\_confidence} < 0.5$,
the tool emits an \texttt{UnrecognizedPatternWarning} listing the
unrecognized operations, alerting developers that the analysis is
incomplete and manual review or a more powerful tool (e.g., Dartagnan)
is needed.

\section{DSL-to-\texttt{.cat} Correspondence}
\label{app:dsl-cat}

We establish a formal correspondence between the \litmus{} DSL
(Section~\ref{sec:additional}) and the \texttt{.cat} specification
language used by herd7~\cite{Alglave2014}.  We demonstrate this
correspondence concretely for x86-TSO, the simplest non-trivial model.

\subsection{The \texttt{.cat} Framework}

A herd7 \texttt{.cat} file defines a memory model as constraints over
execution graphs.  The core constructs are:
\begin{itemize}[nosep]
\item \texttt{let} \textit{relation} \texttt{=} \textit{expr}:
  Define a derived relation (e.g., \texttt{let com = rf | co | fr}).
\item \texttt{acyclic} \textit{relation} \texttt{as} \textit{name}:
  Assert that a relation is acyclic.
\item \texttt{irreflexive} \textit{relation} \texttt{as} \textit{name}:
  Assert that a relation is irreflexive.
\end{itemize}

The x86-TSO \texttt{.cat} model (simplified from herd7) is:
\begin{lstlisting}[caption={x86-TSO.cat (simplified)}]
let com = rf | co | fr
let po-loc = po & loc            (* same-address po *)
let implied = po-loc | com | mfence | [R];po;[R] | [R];po;[W] | [W];po;[W]
acyclic implied as tso
\end{lstlisting}

This states: the global happens-before relation (combining communication
edges, same-address po, and all preserved po edges) must be acyclic.
Under TSO, only store$\to$load reordering to different addresses is
relaxed; all other program-order edges are preserved.

\subsection{DSL-to-\texttt{.cat} Mapping}

Our DSL encodes models as:
\begin{lstlisting}[caption={x86-TSO in \litmus{} DSL}]
model TSO {
    relaxes W->R
    fence mfence { orders R->R, R->W, W->R, W->W }
}
\end{lstlisting}

\begin{proposition}[DSL-\texttt{.cat} Correspondence for x86-TSO]
\label{prop:dsl-cat}
The \litmus{} DSL definition of TSO and the herd7 x86-TSO\texttt{.cat}
specification accept exactly the same set of consistent executions for
any litmus test with finitely many memory operations and no control
flow.
\end{proposition}

\begin{proof}[Proof sketch]
We show that the preserved program order ($\mathit{ppo}$) edges computed
by the DSL engine coincide with those specified by the \texttt{.cat} file.

\textbf{(1)~po-loc:} Both preserve all same-address program-order edges.
The DSL engine always includes po-loc edges (line~488 of \texttt{portcheck.py}:
``if $a.\mathit{addr} = b.\mathit{addr}$: include = True'').
The \texttt{.cat} file includes \texttt{po-loc} in the acyclicity check.

\textbf{(2)~Relaxed pairs:} The DSL declares \texttt{relaxes W->R},
meaning only store$\to$load edges to different addresses are dropped
from $\mathit{ppo}$.  The \texttt{.cat} file achieves the same effect
by listing all \emph{preserved} edges: \texttt{[R];po;[R]},
\texttt{[R];po;[W]}, \texttt{[W];po;[W]}---exactly the complement of
store$\to$load.

\textbf{(3)~Fence semantics:} The DSL declares a fence \texttt{mfence}
that orders all four pair types.  The \texttt{.cat} file includes
\texttt{mfence} edges in the acyclicity check, which has the same
effect: an \texttt{mfence} between a store and a subsequent load
restores the dropped W$\to$R edge.

\textbf{(4)~Communication:} Both derive the same communication edges
($\mathit{rf} \cup \mathit{co} \cup \mathit{fr}$) from the execution
graph.  The DSL engine constructs these in the $\mathit{RF} \times
\mathit{CO}$ enumeration; the \texttt{.cat} file defines
\texttt{com = rf | co | fr}.

\textbf{(5)~Acyclicity:} Both check acyclicity of
$\mathit{ppo} \cup \mathit{com}$.  The DSL engine uses DFS cycle
detection on the ghb graph; the \texttt{.cat} file uses the
\texttt{acyclic} directive.

Since the edge sets and acyclicity checks coincide, the two
specifications accept identical execution sets.
\end{proof}

\begin{remark}[Scope of correspondence]
Proposition~\ref{prop:dsl-cat} establishes the correspondence for x86-TSO.
We have extended the validation empirically to all three CPU models:
the DSL definitions for AArch64 and RISC-V achieve 100\% and 98.2\%
behavioral agreement respectively across all 57 CPU patterns
(Section~\ref{sec:dsl-cat}).  All three models pass structural
correspondence checks (relaxed pairs, dependency preservation,
multi-copy atomicity).  The single mismatch (\texttt{mp\_fence\_wr}
on RISC-V) arises from the DSL's inability to express asymmetric
predecessor/successor fence sets---a known expressiveness gap that does
not affect the built-in checker.  The DSL does not currently support
the \texttt{.cat} constructs \texttt{let rec} (recursive relations),
\texttt{try}/\texttt{with} (optional relations), or user-defined
transitive closures.
\end{remark}


\section{Compositional Reasoning}
\label{app:compositional}

\litmus{} analyzes patterns in isolation.  A natural question is:
when do pattern-level safety guarantees compose into program-level
guarantees?  We formalize sufficient conditions for safe composition
and explain why unrestricted composition is undecidable.

\begin{definition}[Pattern Composition]
Given two patterns $T_1$ and $T_2$ with disjoint forbidden outcomes
$F_{T_1}$ and $F_{T_2}$, their \emph{parallel composition}
$T_1 \| T_2$ is the concurrent program executing all threads of
$T_1$ and $T_2$ simultaneously.
\end{definition}

\begin{theorem}[Disjoint-Variable Composition]
\label{thm:composition}
If patterns $T_1$ and $T_2$ have \emph{disjoint variable sets}
($\mathit{vars}(T_1) \cap \mathit{vars}(T_2) = \emptyset$), and both
are safe under model $M$, then $T_1 \| T_2$ is safe under $M$ with
respect to both $F_{T_1}$ and $F_{T_2}$.
\end{theorem}

\begin{proof}
Since $T_1$ and $T_2$ share no variables, no cross-pattern communication
edges ($\mathit{rf}$, $\mathit{co}$, $\mathit{fr}$) exist between their
operations in any execution of $T_1 \| T_2$.  The ghb graph decomposes
into two independent subgraphs, one for each pattern.  An execution of
$T_1 \| T_2$ is consistent under $M$ iff the projections to $T_1$ and
$T_2$ are each consistent under $M$.  Since $T_1$ is safe (no consistent
execution produces $F_{T_1}$) and $T_2$ is safe (no consistent execution
produces $F_{T_2}$), neither forbidden outcome is producible in
$T_1 \| T_2$.
\end{proof}

\begin{proposition}[Shared-Variable Composition is Unsafe in General]
\label{prop:unsafe-composition}
There exist patterns $T_1, T_2$ that are each safe under ARM, but
whose composition $T_1 \| T_2$ is unsafe when $T_1$ and $T_2$ share
a variable.
\end{proposition}

\begin{proof}[Proof by example]
Let $T_1$ be an MP-fence pattern (safe on ARM) using variables $x, y$
with a fence between \texttt{St~x} and \texttt{St~y}.  Let $T_2$ be
a separate MP-fence pattern (also safe on ARM) using variables $y, z$
with a fence between \texttt{St~y} and \texttt{St~z}.  Both $T_1$ and
$T_2$ are individually safe.

In the composition, thread~0 of $T_1$ writes $x$ then $y$ (with fence);
thread~0 of $T_2$ writes $y$ then $z$ (with fence); and reader threads
observe $y, x$ and $z, y$ respectively.  The shared variable $y$
creates cross-pattern coherence-order edges that can combine with the
readers' program-order edges to form cycles that were impossible in
either pattern alone, potentially making the composed program unsafe.
\end{proof}

\begin{remark}[Practical Implications]
Theorem~\ref{thm:composition} provides a sound over-approximation
for programs that can be decomposed into patterns with disjoint
variable sets---a common case in well-designed concurrent programs
where each synchronization idiom guards distinct data.

For programs with shared variables across synchronization patterns,
\litmus{} results should be interpreted as necessary conditions
(each pattern must be individually safe) but not sufficient.
Full-program tools (Dartagnan~\cite{Gavrilenko2019},
VSync~\cite{Oberhauser2021}) are needed for compositional verification.

The general problem of deciding whether a program composed of
individually safe patterns is itself safe is at least as hard as
the reachability problem for concurrent programs, which is
undecidable for unbounded thread creation and PSPACE-complete for
bounded concurrent programs.
\end{remark}


\section{Severity Classification Details}
\label{app:severity-details}

The 342 unsafe (pattern, architecture) pairs are classified by severity
based on the violated memory ordering and its consequences:

\begin{itemize}[nosep]
\item \textbf{Security vulnerability (44 pairs):} Patterns implementing
  mutual exclusion (Dekker, Peterson) where relaxed ordering breaks
  atomicity invariants, and GPU scope mismatch patterns where
  insufficient synchronization scope allows cross-workgroup data races.
  These require immediate remediation.

\item \textbf{Data race (228 pairs):} Standard weak-memory hazards where
  relaxed store$\to$load, store$\to$store, or load$\to$load ordering
  allows stale or inconsistent reads. Subcategories include message passing
  (mp variants), store buffering (sb variants), load buffering (lb variants),
  multi-copy atomicity violations (iriw), and causality patterns
  (wrc, rwc, isa2, r, s). Most are fixable with appropriate fences.

\item \textbf{Benign (70 pairs):} Coherence-order artifacts (CoWR, CoWW, 2+2W)
  that are architecturally allowed but do not affect data integrity for
  programs with single-writer semantics. These represent ordering visibility
  differences rather than correctness bugs.
\end{itemize}

Per-architecture severity distribution:
\begin{itemize}[nosep]
\item x86 (TSO): fewest unsafe pairs, dominated by store-buffering data races
\item ARM/RISC-V: most unsafe pairs (maximal relaxation), broad data-race distribution
\item GPU models: concentrated security vulnerabilities from scope mismatches
\end{itemize}
