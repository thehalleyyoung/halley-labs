## Approach A: "Noether's Scalpel" — Full Static Symbolic Pipeline

### Strategy

Pure static analysis from source to verdict. Parse Python simulation code via Tree-sitter, lift numerical kernels into a conservation-aware IR (polynomial-rational maps on phase spaces), run restricted Lie-symmetry analysis to identify continuous symmetries, compute provenance-tagged modified Hamiltonians via BCH expansion, and perform differential symbolic slicing to trace symmetry-breaking terms to source lines. No execution of user code at any point. The tool is a compiler from imperative simulation code to conservation diagnostics.

### 1. Extreme Value Delivered and Who Desperately Needs It

**Who:** Numerical methods researchers implementing structure-preserving integrators in Python (JAX-MD, Dedalus, SciPy). A PhD student publishes a "symplectic, energy-conserving" integrator for molecular dynamics. Three years later someone discovers it silently leaks angular momentum due to an asymmetric force splitting — the exact scenario from Wan et al. (2019, JAMES). ConservationLint catches this on the first commit.

**Pain:** Today, verifying conservation properties requires a domain expert to manually reconstruct the modified equation on paper, then cross-reference against the code line by line. This takes days per integrator and scales to zero for multi-method compositions. No tool provides causal localization ("line 247 breaks rotational equivariance") or obstruction detection ("this splitting *cannot* conserve angular momentum at any order — restructure"). Existing monitors (GROMACS `gmx energy`, LAMMPS `thermo_style`) detect *that* energy drifts; they cannot say *why* or *where*, and they miss momentum/angular momentum/enstrophy entirely unless manually configured.

**Value:** Transforms conservation verification from a weeks-long expert ritual into a 10-minute CI check with formal guarantees. The obstruction detector (T2) is uniquely valuable: it saves developers from futile debugging by proving that certain violations are architecturally unfixable without restructuring the splitting.

### 2. Genuine Difficulty as a Software Artifact

The make-or-break engineering challenge is **code→math extraction**: lifting imperative NumPy code with broadcasting, aliasing, mutation, and library calls into a faithful polynomial-rational IR. This is not parsing (Tree-sitter handles syntax trivially) but *semantic lifting* — recognizing that `np.add.at(forces, idx, -grad_potential)` implements force accumulation on a particle system. Every NumPy/SciPy function needs a handwritten mathematical model. Pattern lifting is inherently brittle: the same Verlet step can be written in 20 syntactic variants.

The **integration risk** across subsystems is severe. The IR must be simultaneously consumable by the Lie-symmetry analyzer (needs group action structure), the BCH engine (needs operator splitting structure), and the localizer (needs source-line provenance). A single abstraction mismatch kills the pipeline.

**Novel algorithms required:** (a) Differential symbolic slicing — program slicing where the criterion is "contribution to order-n term in a formal power series," not data dependence. (b) Provenance-tagged BCH — standard BCH with operator labels propagated through each Lie bracket, requiring careful bookkeeping in a non-commutative free algebra. (c) Obstruction ideal computation — checking whether a polynomial ideal generated by Lie bracket expressions is trivial, using Gröbner basis methods restricted to the BCH-induced structure.

### 3. New Math Required

**T1: Provenance-Tagged Heterogeneous Composition Modified Equation.** Generalizes classical backward error analysis from single-method integrators to k-way compositions of methods with *different* orders p_i. The tagged structure is not mere bookkeeping in the mixed-order case: when sub-integrators have different orders, cross-terms in the BCH expansion create interaction effects absent from homogeneous compositions. The theorem must establish that each correction term δ_n in H̃ = H + Σ h^n δ_n decomposes as a sum of tagged Lie monomials, each carrying the subset of sub-integrators that generate it. **Load-bearing because:** without provenance tags, localization degrades to "somewhere in the integrator" — no better than existing monitors.

**T2: Computable Obstruction Criterion.** Reduces "does there exist *any* composition of methods (of given orders) for this splitting that preserves conservation law C to order p?" to a finite algebraic test. The non-trivial content is exploiting BCH structure to avoid doubly-exponential quantifier elimination (Tarski-Seidenberg). Must yield polynomial-time complexity for fixed k, p. **Load-bearing because:** without it, every detected violation suggests a local fix, many of which are provably futile.

### 4. Best-Paper Potential

The "impossible bridge" narrative — connecting geometric numerical integration (which knows everything about conservation but works on paper) with program analysis (which knows everything about code but has never targeted physical invariants) — is genuinely novel. No prior work operates in the code→math direction for conservation. If T2 yields an elegant structural result (e.g., "a splitting obstructs conservation of angular momentum iff the commutator [H_i, H_j] has a non-vanishing component along the rotation generator for some i≠j"), this is a beautiful theorem with immediate practical impact. The "killer demo" — automatically rediscovering the Wan et al. climate-model conservation bug — would be a showstopper. **Target venues:** OOPSLA (domain-specific tool + evaluation), CAV (formal guarantees + T2), PLDI (differential symbolic slicing as novel PL technique).

### 5. Hardest Technical Challenge

**Code→math extraction on real code.** JAX traces Python to build jaxprs — invisible to Tree-sitter. NumPy broadcasting requires shape information unavailable statically. SciPy sparse solvers are opaque. **Mitigation:** (a) Restrict Phase 1 to pure-NumPy code with explicit loops (the "liftable fragment") — this covers textbook integrators and research prototypes. (b) Build a semantic database of 200–300 NumPy/SciPy function signatures with mathematical models. (c) For JAX, intercept at the jaxpr level (JAX exposes its trace IR) rather than parsing Python source. (d) Accept 20–40% coverage and position as formal verification for the analyzable fragment, with explicit fallback recommendations for code outside it. If coverage falls below 15% on 3/5 real codebases, abandon.

### 6. Scores

| Axis | Score | Rationale |
|------|-------|-----------|
| **Value** | 6/10 | Uniquely solves localization + obstruction for formal-methods-demanding users; narrow TAM (~5K developers) |
| **Difficulty** | 7/10 | Code→math extraction is genuinely unprecedented; integration across subsystems is high-risk |
| **Potential** | 6/10 | Strong narrative, but T1 may be bookkeeping and coverage may be too low for impressive evaluation |
| **Feasibility** | 5/10 | Existential extraction risk; 40% chance of Phase 1 failure leading to abandon |

## Approach B: "Conservation Types" — Graded Effect System for Symmetry-Aware Composition

### Strategy

Instead of analyzing arbitrary code *post hoc*, design a **type system** where conservation laws are first-class types and integrator composition is type-checked. Numerical operations carry *symmetry grades* — annotations from the Lie algebra of the physical system — and composition of operations is governed by a graded monoid that tracks which symmetries survive. A well-typed program is *guaranteed* to preserve declared conservation laws. Ill-typed programs get precise error messages: "composing your O(Δt²) symplectic step with this thermostat breaks the energy grade — the composed type is Grade(energy: broken@O(Δt¹), momentum: preserved, angular_momentum: preserved@O(Δt²))."

The tool ships as: (1) a Python library of typed numerical primitives (integrator steps, force evaluators, splitters) with symmetry-grade annotations, (2) a lightweight static checker (not a full compiler — closer to mypy for conservation) that verifies grade consistency at composition boundaries, and (3) a grade inference engine that automatically computes the conservation grade of user-defined numerical kernels via abstract interpretation over the symmetry-grade lattice.

### 1. Extreme Value Delivered and Who Desperately Needs It

**Who:** Framework developers building simulation infrastructure (JAX-MD maintainers, Dedalus core team, OpenMM plugin authors). These developers compose numerical methods as building blocks — a symplectic integrator here, a thermostat there, an Ewald summation for long-range forces. They need to know *at composition time* whether the assembly preserves the intended conservation laws, not after running a million-step simulation and observing drift.

**Pain:** Current workflow: build the integrator, run a long simulation, plot energy vs. time, discover it drifts, manually debug which component broke it. This takes days and catches only energy — momentum and angular momentum violations go unnoticed until a user reports unphysical behavior. With conservation types, the violation is caught *at integration time* (not simulation time), and the type error message pinpoints the incompatible composition.

**Unique value:** Prevention over detection. This is the only approach that makes conservation violations *impossible* for well-typed programs, analogous to how Rust's borrow checker makes use-after-free impossible. The graded structure provides a *compositional* analysis: each component is typed independently, and composition correctness follows from type compatibility — no whole-program analysis needed.

### 2. Genuine Difficulty as a Software Artifact

**Designing the grade lattice.** Conservation is not binary; it's graded by order. A symplectic integrator preserves a *modified* Hamiltonian to all orders but the *true* Hamiltonian only to order p. The grade lattice must capture: (a) which conservation laws are preserved, (b) to what order, (c) with what leading error coefficient structure. This is a non-trivial algebraic structure — a graded commutative monoid indexed by the Lie algebra of the symmetry group, with order annotations forming a tropical semiring.

**Grade inference is abstract interpretation.** For user-defined kernels (not just library primitives), the checker must infer conservation grades by abstract interpretation: propagating symmetry-grade information through arithmetic operations, array manipulations, and control flow. The abstract domain is the lattice of symmetry grades; transfer functions model how each operation affects conservation. Widening is needed for loops. This is a non-standard abstract domain with no prior art.

**Adoption barrier.** Unlike Approach A (which analyzes existing code), this requires users to either (a) use the typed primitive library or (b) submit to grade inference on their existing code. Inference on arbitrary NumPy code faces the same extraction challenges as Approach A; the type system's value is maximized when users opt into the typed API.

### 3. New Math Required

**Graded Monoidal Conservation Type Theory.** Define a type system where types are annotated with elements of a *conservation grade monoid* G = (Sym × ℕ∞)^k, where Sym is the symmetry group, ℕ∞ tracks preservation order, and k indexes the number of independent conservation laws. Integrator composition corresponds to the monoid operation, which combines grades via the BCH formula truncated at the relevant order. The key theorem: **Soundness** — if a program type-checks with grade g, then the implemented map preserves the conservation laws indicated by g to the indicated order. **Completeness (restricted)** — for programs built from typed primitives, grade inference computes the tightest possible grade.

**Connection Theorem: Lie Algebra → Effect Grading.** The conservation grade monoid is isomorphic to a quotient of the free Lie algebra on the sub-Hamiltonian generators, modulo the ideal of terms that vanish under the Noether pairing with the conserved quantity. This connects the PL concept (effect grading) with the physics concept (Lie-algebraic structure of conservation) in a precise way. **Load-bearing because:** without this connection, the grade lattice is ad hoc; with it, the type system inherits the full structure of BCH theory, making grade computation efficient and mathematically justified.

### 4. Best-Paper Potential

This approach has the strongest *PL-theoretic* contribution. The connection between Lie algebras and graded effect systems is genuinely novel and would interest the POPL/ICFP community beyond the scientific computing audience. "Conservation laws as types" is a one-sentence pitch that resonates across communities. The soundness theorem gives a formal guarantee no other approach provides — well-typed programs *cannot* violate conservation. The paper could demonstrate the type system on 10–15 integrator compositions drawn from real frameworks, showing that type errors correspond exactly to known conservation bugs.

**The "why this wins best paper" argument:** It's a clean, beautiful idea with a precise theorem connecting two deep theories (Lie symmetry ↔ graded effects). Unlike Approach A's sprawling pipeline, the core contribution fits in 12 pages. If the Connection Theorem yields surprises (e.g., "the grade lattice for Hamiltonian systems is always a finite distributive lattice, and its Möbius function computes the obstruction"), this is a landmark result. **Target venues:** POPL (type theory + semantics), ICFP (graded effects), OOPSLA (practical type system for domain).

### 5. Hardest Technical Challenge

**Grade inference on existing code without typed annotations.** If users must rewrite their code using typed primitives, adoption will be near-zero. The tool must infer conservation grades for arbitrary NumPy/SciPy code — and this requires solving the same code→math extraction problem as Approach A. **Mitigation:** (a) Provide a two-tier experience: *typed API* for new code (full guarantees, zero extraction risk) and *grade inference* for existing code (best-effort, same extraction challenges). (b) Partner with framework developers to annotate JAX-MD/Dedalus primitives with conservation grades, making the typed API the default for new code. (c) Accept that inference on arbitrary code will have the same 20–40% coverage as Approach A; the unique value is the *compositional* guarantee for code built from typed primitives. The risk is that the typed API is too restrictive or too different from existing APIs for adoption.

### 6. Scores

| Axis | Score | Rationale |
|------|-------|-----------|
| **Value** | 5/10 | Elegant prevention-by-construction, but requires API adoption; lower immediate value than post-hoc analysis |
| **Difficulty** | 8/10 | Novel abstract domain, graded type theory, Lie-algebra–effects connection; no prior art for the grade lattice |
| **Potential** | 8/10 | Strongest theoretical contribution; Lie↔effects connection is publishable at POPL independent of tool |
| **Feasibility** | 4/10 | Adoption barrier is severe; grade inference on existing code inherits all extraction risks; may produce a beautiful theory with no users |

## Approach C: "Shadow Oracle" — Dynamic Backward Error Analysis via Instrumented Execution

### Strategy

Abandon the fragile code→math extraction entirely. Instead, **execute** the simulation code with lightweight instrumentation, collect phase-space trajectories, and use **sparse symbolic regression** to recover the *shadow Hamiltonian* — the modified Hamiltonian that the integrator actually conserves. Compare the recovered shadow Hamiltonian against the declared/intended Hamiltonian to identify conservation violations. Localize violations by **ablation**: systematically toggling code regions (force terms, splitting stages) on and off, re-running short trajectories, and measuring the change in the symmetry-breaking terms of the shadow Hamiltonian. This is a fully dynamic approach that works on *any runnable Python code* — JAX, opaque library calls, FFT-based spectral methods, GPU kernels — with zero extraction.

The tool ships as: (1) a Python decorator `@conservation_audit` that instruments a simulation's time-stepper, (2) a shadow Hamiltonian recovery engine using SINDy-style sparse regression on phase-space data, (3) a symmetry tester that checks recovered quantities against declared symmetry generators, and (4) an ablation-based localizer that attributes violations to code regions via causal intervention.

### 1. Extreme Value Delivered and Who Desperately Needs It

**Who:** *Every* Python simulation developer, regardless of code complexity or framework. A Dedalus user writing a spectral PDE solver with FFT-based operators. A JAX-MD user composing `jit`/`vmap`-compiled force computations. A SciPy user calling `solve_ivp` with a black-box right-hand side. None of these can be statically analyzed by Approach A (opaque libraries, traced computation graphs, FFT). All of them can be *run*.

**Pain:** The same as the other approaches — silent conservation violations — but the key differentiator is **coverage**. Approach A covers 20–40% of kernel code; Shadow Oracle covers 100% of *runnable* code. For the 60–80% of code that falls outside the liftable fragment, there is currently *no* automated conservation analysis at all. The tradeoff is statistical guarantees (from finite traces) rather than formal proofs, but statistical guarantees on 100% of code beat formal proofs on 20%.

**Unique value:** (a) Works on any runnable Python simulation — no liftable fragment restriction. (b) Automatically discovers conservation laws the user didn't declare, via symmetry scanning of the recovered shadow Hamiltonian. (c) The ablation-based localizer works even when the code is opaque (just toggle regions and measure the effect). (d) Provides *quantitative* conservation error estimates (not just "violated" but "energy drifts at rate 3.7×10⁻⁴ per step, dominated by the thermostat term").

### 2. Genuine Difficulty as a Software Artifact

**Shadow Hamiltonian recovery from finite, noisy traces.** The integrator conserves a modified Hamiltonian H̃ = H + h²H₃ + h⁴H₅ + ..., where the correction terms are unknown. Recovering H̃ from trajectory data requires solving a sparse regression problem in the space of polynomial expressions of the phase-space variables. The dictionary must include all monomials up to a chosen degree — the combinatorial explosion for >10 state variables is severe. For a 100-particle molecular dynamics system (600-dimensional phase space), naive polynomial regression is intractable.

**Exploiting physical structure in the regression.** The regression must be regularized using physical priors: (a) H̃ is a sum of few-body interactions (sparse in particle pairs, not in full phase-space monomials), (b) each correction term H_{2k+1} has a known scaling with h, enabling multi-scale regression across different step sizes, (c) the terms are Lie polynomials in the sub-Hamiltonians, constraining the functional form. Incorporating these priors into the regression is a novel algorithmic challenge.

**Ablation-based localization requires controlled execution.** The localizer must systematically disable code regions and measure the effect on conservation. This requires: (a) automatic identification of "toggleable" code regions (force terms, splitting stages), (b) a mechanism to disable them (mocking, zeroing, or bypassing), and (c) enough short-trajectory samples to reliably attribute conservation changes to specific regions via causal inference (essentially, a designed experiment on code). The statistical power required to distinguish "this region contributes ε to the symmetry-breaking" from noise is non-trivial.

**No formal obstruction detection.** Unlike Approach A, the dynamic approach cannot prove that a violation is architecturally unfixable — it can only measure the violation's magnitude and localize it. The T2 obstruction criterion requires algebraic analysis of the splitting structure, which the dynamic approach doesn't perform. This is a genuine capability gap.

### 3. New Math Required

**Convergence Theorem for Shadow Hamiltonian Recovery.** Given N trajectory points from an integrator of order p applied to a Hamiltonian system with d degrees of freedom, the sparse regression recovery of the shadow Hamiltonian H̃ truncated at order q satisfies ‖Ĥ̃ - H̃‖ ≤ C(d,q) · N^{-α} · h^{p+q+1} with probability ≥ 1-δ, where α depends on the regression method and C(d,q) is explicit. The key challenge: establishing the *restricted isometry property* (RIP) for the physical-prior-constrained dictionary — i.e., proving that trajectory data from Hamiltonian systems provides enough "diversity" to recover the shadow Hamiltonian uniquely. **Load-bearing because:** without convergence guarantees, the tool's output is heuristic and unreliable; with them, users get confidence intervals on every reported conservation violation.

**Causal Ablation Sensitivity Theorem.** When a code region contributing term δH to the Hamiltonian is disabled, the change in the shadow Hamiltonian's symmetry-breaking component is Δ(symmetry break) = {δH, C_v} + O(h²), where C_v is the conserved quantity. This first-order sensitivity is exact for additive splittings and approximate for more general compositions. **Load-bearing because:** it validates that ablation-based localization correctly identifies the *causal* contributor to a conservation violation, not merely a correlated one.

### 4. Best-Paper Potential

The "100% coverage" claim is the killer feature. If the evaluation shows that Shadow Oracle detects conservation violations in JAX-MD, Dedalus, and SciPy codes that Approach A cannot even parse, and does so with <5% false positive rate and quantitative error estimates, this is a compelling result. The paper demonstrates a fundamentally different philosophy: instead of fighting the code→math extraction problem, side-step it entirely by treating the code as a black box and recovering the math from execution traces. This is reminiscent of how fuzzing succeeded where static analysis couldn't — by embracing dynamic execution.

**The contrarian narrative:** "Symbolic static analysis of simulation code is the wrong approach. The code is too complex, the abstractions too opaque, the coverage too low. But the *trajectories* the code produces are rich, structured, and physically constrained. Recovery of the shadow Hamiltonian from trajectories is the right level of abstraction for conservation analysis." If the convergence theorem is tight and the ablation localizer is precise, this challenges the conventional wisdom that formal methods require static analysis. **Target venues:** ICSE (practical tool + evaluation), NeurIPS (sparse recovery + scientific ML angle), SC (scientific computing community), OOPSLA (novel dynamic program analysis).

### 5. Hardest Technical Challenge

**Scalability of shadow Hamiltonian recovery to high-dimensional systems.** A 100-particle 3D molecular dynamics system has 600 state variables. Even with physical priors (pairwise interactions, translation invariance), the dictionary of candidate terms grows combinatorially. **Mitigation:** (a) Exploit particle permutation symmetry to reduce the dictionary to *types* of interactions (pair potential, three-body potential) rather than enumerating all particle pairs. (b) Use the multi-scale regression trick: run at 3–5 different step sizes h, and the h-scaling of each correction term disambiguates contributions at different orders. (c) For truly high-dimensional systems, use a two-stage approach: first detect *which* conservation laws are violated (a lower-dimensional problem — check d/dt of each conserved quantity along the trajectory), then recover only the symmetry-breaking component of the shadow Hamiltonian (a much sparser problem than full recovery). (d) Accept that full shadow Hamiltonian recovery is impractical for >50 particles; for larger systems, fall back to conservation-law-specific violation detection without full localization. The risk is that the regression is too noisy for reliable localization in high-dimensional systems, yielding a tool that detects but cannot localize in the most important use cases.

### Recommended Strategy

**Build C first, then A.** Shadow Oracle (C) has the highest feasibility and broadest value — ship it as the default mode that works on everything. In parallel, develop Noether's Scalpel (A) for the liftable fragment, providing formal guarantees where they're achievable. The two approaches are complementary: C detects and localizes dynamically on 100% of code; A provides formal proofs and obstruction detection on the 20–40% that admits static analysis. The combined tool offers a graceful degradation: formal guarantees where possible, statistical guarantees everywhere else.

Conservation Types (B) is the highest-risk, highest-ceiling approach. If the Lie↔effects connection theorem works out, it's independently publishable at POPL and could become the long-term foundation for a new generation of physics-aware type systems. But it requires framework adoption and may never escape the typed-API niche. **Recommend:** pursue the Connection Theorem as a theory paper in parallel, but don't bet the tool on it.
