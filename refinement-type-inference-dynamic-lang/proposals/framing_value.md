# Zero-Annotation Refinement Type Inference That Makes Python and TypeScript as Safe as Ada—Incrementally, in CI, on a Laptop

## Problem and Approach

Dynamic languages won the adoption war. Python dominates ML infrastructure, scientific computing, and backend services; TypeScript dominates the web. Together they underpin the software that runs hospitals, power grids, and financial markets. Yet these codebases ship with a class of defects that statically-typed languages eliminated decades ago: out-of-bounds array access, null dereference, division by zero, and type-tag confusion. These aren't exotic bugs—they are the **top four categories** in CVE databases for web applications and the leading cause of silent data corruption in numerical Python pipelines. Existing mitigations (unit tests, mypy, tsc strict mode) catch surface-level type errors but are structurally blind to value-dependent properties: *is this index within bounds? is this divisor provably nonzero after that branch?* Developers know this. They write ad-hoc assertions, defensive checks, and pray. The gap between what the language lets you express and what the toolchain can verify is where production incidents live.

We propose a system that **automatically infers refinement types**—predicates over program values such as `{x: int | x > 0 ∧ x < len(arr)}`—for Python and TypeScript codebases, requiring zero developer annotation. The engine combines abstract interpretation to discover candidate invariants with SMT-backed predicate inference in the style of Liquid Types, using counterexample-guided abstraction refinement (CEGAR) to iteratively strengthen predicates when initial templates are insufficient. The output is a set of **per-function, machine-checkable contracts** that certify array bounds safety, null/undefined safety, division-by-zero freedom, and type-tag correctness. Crucially, the analysis is **incremental**: on each commit, only changed functions and their transitive dependents are re-analyzed, making the system practical as a CI gate rather than a research artifact that runs overnight. The entire system targets commodity laptop CPUs—no GPU, no cloud cluster, no human in the loop.

The key architectural insight is that dynamic language idioms (duck typing, runtime type tests, dictionary-as-struct patterns) are not obstacles to refinement typing but rather *sources of rich predicates* that static languages lack. A runtime `isinstance` check in Python or a `typeof` guard in TypeScript is an explicit predicate that the developer already wrote; our system harvests these as seeds for the CEGAR loop, dramatically reducing the predicate search space compared to inferring refinements for languages where such checks are implicit in the type system.

## Value Proposition: Who Needs This, Why Desperately

**ML infrastructure teams** are the acute case. A misindexed tensor dimension in a training pipeline doesn't crash—it silently produces wrong gradients and a model that underperforms for weeks before anyone notices. Facebook, Google, and every ML startup have war stories of shape-mismatch bugs that cost GPU-months. These teams write Python, they cannot switch languages, and mypy doesn't understand `tensor[batch, seq_len, hidden]`. Refinement types over array dimensions are the *only* static technique that can catch these errors pre-merge.

**Financial technology and healthcare software** written in TypeScript/Python faces regulatory pressure (SOC 2, HIPAA, FDA software validation) to demonstrate absence of specific defect classes. Today, compliance teams pay for manual code audits or expensive commercial static analyzers that produce overwhelming false-positive reports. A tool that emits machine-checkable certificates of bounds safety and null safety, integrated into CI, replaces a manual audit with a cryptographically-meaningful artifact.

**Open-source library maintainers** (NumPy, Pandas, Express, Next.js) field thousands of bug reports per year rooted in misuse of value-dependent APIs. Inferred refinement type signatures would serve as **executable documentation**: not just "this function takes an int" but "this function takes an int that must be between 0 and len(self.columns)." This collapses the gap between documentation, types, and runtime checks into a single verified artifact.

## Why It's Genuinely Hard to Build

1. **Predicate discovery at scale is an open problem.** Liquid Types work brilliantly when the programmer supplies predicate templates. Inferring the right predicates automatically—especially across dynamic dispatch, higher-order functions, and Python's mutable default arguments—requires a CEGAR loop that converges in reasonable time. No existing system has demonstrated this for real-world Python or TypeScript codebases beyond toy benchmarks.

2. **Incrementality is a systems-level nightmare.** Sound incremental re-analysis requires precise dependency tracking through dynamic features: monkey-patching, prototype chain mutation, `**kwargs` forwarding, and decorator wrapping. A naïve "re-analyze everything downstream" strategy collapses to whole-program analysis on real codebases. The dependency graph itself must be refined incrementally.

3. **Two languages, two semantics, one framework.** Python and TypeScript have deeply different evaluation strategies, scoping rules, and object models. Unifying them under a single abstract interpretation framework without sacrificing precision in either language demands a carefully designed intermediate representation that preserves language-specific refinement-relevant semantics (e.g., Python's truthiness coercions vs. TypeScript's strict equality).

4. **False-positive rate is existential.** A refinement type system that produces even 5% false alarms on real codebases will be disabled in CI within a week. Achieving near-zero false positives while maintaining meaningful coverage requires the SMT encoding to faithfully model standard library semantics (len, slice, dict.get, Array.prototype.map) at a fidelity that no current tool achieves.

5. **~165K LoC of implementation.** This is a compiler-scale engineering effort, not a weekend prototype. The system must include parsers, a shared IR, abstract domains, SMT integration, CEGAR orchestration, incremental dependency tracking, CI integration, and human-readable error reporting—each individually a publishable contribution.

## Best-Paper Argument

This paper would earn a best-paper award because it **collapses a 20-year gap between formal methods theory and developer practice** in a way no prior work has achieved. Liquid Types (Rondon et al., PLDI 2008) showed refinement type inference was possible; this work shows it is *practical on the languages developers actually use, at the scale they actually work, in the workflow they actually have*. The evaluation story is compelling and fully automated: run the tool on the 100 most-starred Python and TypeScript repositories, measure (a) bugs found that existing tools miss, (b) false-positive rate, (c) incremental re-analysis time per commit, (d) CEGAR convergence rate. If the system finds real CVE-class bugs in NumPy or Express with a false-positive rate under 1% and per-commit analysis under 60 seconds, the result is undeniable. The paper bridges PL theory (refinement types, abstract interpretation, CEGAR) and SE practice (CI integration, incremental analysis) in a way that both PLDI and ICSE/FSE reviewers would champion. It is the rare paper that is simultaneously a theoretical contribution (automatic predicate inference for dynamic languages), a systems contribution (incremental analysis at scale), and an empirical contribution (large-scale bug finding on real-world code).

## Fatal Flaws to Watch

1. **The false-positive cliff.** If the system cannot achieve <2% false positives on real codebases, the entire CI-integration narrative collapses. Dynamic language idioms (eval, runtime metaprogramming, C extensions) may force the system to either over-approximate (false alarms) or under-approximate (missed bugs), and it's unclear which regime real-world code falls into. **Mitigation:** scope the soundness guarantee to a well-defined "analyzable subset" and measure coverage honestly.

2. **CEGAR non-convergence on real code.** CEGAR loops can diverge or take exponential iterations on programs with complex control flow. If the tool times out on 30% of functions in popular repositories, the "practical for CI" claim is dead. **Mitigation:** set hard time budgets per function and report "unknown" rather than unsound results; measure the unknown rate explicitly.

3. **Standard library modeling is a bottomless pit.** Python's standard library is enormous and semantically complex (e.g., `itertools`, `collections.abc`, NumPy's broadcasting rules). The system's utility is bounded by how much of the standard library it can model. If it can only reason about `len` and basic arithmetic, the inferred refinements will be too shallow to catch interesting bugs. **Mitigation:** prioritize the 50 most-used standard library functions by call frequency in the target corpus.

4. **Incremental analysis may not be incremental enough.** In highly interconnected codebases (e.g., monorepos with deep import graphs), a single function change may transitively affect thousands of dependents. If median re-analysis time exceeds 5 minutes, teams will not gate CI on it. **Mitigation:** use modular analysis with summary-based function contracts to break transitive chains.

5. **Novelty risk from the PL side.** Reviewers at PLDI/POPL may argue this is "just engineering"—applying known techniques (Liquid Types + CEGAR + abstract interpretation) to new languages without a fundamental theoretical advance. **Mitigation:** the predicate inference algorithm for dynamic language idioms (harvesting runtime type tests as CEGAR seeds) and the incremental re-analysis framework for refinement types are both novel contributions that should be foregrounded in the paper's technical narrative.
