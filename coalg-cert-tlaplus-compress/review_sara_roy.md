# Review: Witness-Certified Coalgebraic Compression for Finite-Domain TLA+ Model Checking

**Reviewer:** Sara Roy (ML + Formal Verification)
**Score: 5/10**

---

From the ML-verification intersection perspective, this proposal is solidly positioned in the formal verification tradition but represents a missed opportunity to incorporate learning-augmented algorithms more deeply. The L*-style coalgebraic learner is, at its core, an automata learning algorithm adapted to the coalgebraic setting—Angluin's L* is a foundational result in computational learning theory, and the Barlocco-Kupke-Rot (FoSSaCS 2019) framework provides the categorical generalization. However, the proposal treats the learner as a fixed algorithm with known convergence bounds, O(n² · k), without exploring whether modern machine learning techniques could improve the learner's practical performance. Specifically, the equivalence oracle—the bounded conformance tester—is the weakest component, and neural-guided equivalence testing has shown significant improvements in related settings. Angluin-style learners with neural network-guided counterexample generation (e.g., Weiss et al., ICML 2018 for DFA extraction; Mayr and Yovine, ML 2018 for hybrid systems) achieve faster convergence in practice by learning to predict likely counterexamples rather than exhaustively testing to depth k. Adapting such techniques to the stuttering-fair functor setting would require training a neural network to predict distinguishing sequences, which is feasible given the finite-domain structure of TLA-lite and the structured nature of the counterexamples (sequences of TLA-lite actions).

The proposal's treatment of the adaptive rank controller in the CME project (the sibling project in the same research program) suggests awareness of ML-augmented algorithms, but CoaCert-TLA itself contains no learning component beyond L*. A natural extension would be a learned heuristic for the L* learner's counterexample processing—when the conformance tester finds a counterexample, the learner must process it to refine the observation table, and the order in which counterexample suffixes are added affects convergence speed. A trained model predicting which suffixes are most informative (analogous to active learning strategies in ML) could reduce the practical query count well below the O(n² · k) worst case. The evaluation plan's differential testing against TLC is methodologically sound but does not evaluate the learning aspects of the system. There is no analysis of learning curves (how quickly does the quotient converge as a function of queries?), no comparison of different counterexample processing strategies, and no measurement of the learner's sample efficiency relative to the theoretical bound. For a paper claiming to instantiate a learning framework, these omissions are significant.

The ML-verification integration is strongest in the Merkle-hashed witness scheme, which addresses a genuine gap in the verification pipeline: the inability to independently verify claimed verification results. From an ML safety perspective, this is valuable—ML systems increasingly depend on formally verified components, and the ability to audit verification results without re-running the computation is essential for deployment in safety-critical settings. However, the witness scheme certifies only the bisimulation relation, not the temporal logic properties verified on the quotient. A complete audit trail would include: (1) the bisimulation witness (provided), (2) a proof that the quotient satisfies the property (not provided—the user must trust the model checker's property evaluation), and (3) a proof that bisimulation preservation implies the original system satisfies the property (provided as a theorem, but not as a machine-checkable proof). The trusted computing base (TCB) analysis is missing: the user must trust the TLA-lite parser, semantic engine, L* learner, and property evaluator, in addition to the Rust compiler and runtime. A formal TCB enumeration with lines-of-code estimates for each trusted component would strengthen the certified verification narrative.
