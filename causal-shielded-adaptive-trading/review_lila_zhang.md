# Review: Causal Identification and Posterior-Predictive Shields for Provably Safe Adaptive Trading

**Reviewer:** Lila Zhang (Symbolic Reasoning & SAT/SMT Solving)
**Score: 6/10**

---

This project's symbolic reasoning component is significantly weaker than its probabilistic and causal-inference components, which is appropriate given the domain but limits the strength of formal guarantees. The verification of bounded LTL safety specifications over the posterior credible set uses symbolic model checking over a finite abstraction—essentially PRISM-style sparse matrix methods applied to the ~1,000-state abstraction with ~10 actions and horizon H = 20. While this is computationally tractable, it is not pushing the boundaries of symbolic reasoning in any meaningful way. The specifications themselves—bounded drawdown, position limits, margin constraints—are expressible as simple bounded safety properties in LTL and do not require the full expressivity of temporal logic; reachability analysis over the finite abstraction would suffice for most of the claimed specifications. The gap between this and genuine SMT-based or SAT-based verification is significant: there is no constraint solving, no theory combination, no quantifier elimination—just sparse matrix-vector multiplication over a small state space.

The more interesting symbolic reasoning challenge lies in the causal identification pipeline, but this is handled via statistical testing (HSIC-based CI tests) rather than symbolic methods. The e-value construction for the Sequential Causal Invariance Test (SCIT) has a proof-theoretic flavor—the product martingale provides anytime validity via Ville's inequality—but this is probabilistic reasoning, not symbolic reasoning per se. The Lean 4 formalization (25K LoC of proofs across 6 hard subproblems) is where the symbolic reasoning substance resides, and this is a major commitment. Mechanizing the PAC-Bayes bound, the shield liveness theorem, and the composition theorem in Lean 4 would be a significant formal verification achievement. However, the problem statement does not describe how the Lean 4 proofs connect to the runtime system: is there an extraction mechanism from Lean 4 to executable code? Are the proofs about idealized mathematical objects or about the actual Python/NumPy implementation? If the proofs are about idealized models and the implementation uses floating-point arithmetic with finite-precision interval computations, the gap between the formalized guarantees and the actual system behavior could be substantial. The interval arithmetic module in the implementation (for independent verification of shield soundness) is a step toward bridging this gap, but without a formal proof that the floating-point interval arithmetic correctly implements the real-arithmetic specification, the verification chain has a weak link at the implementation level.
