# Where in the Pipeline Did It Break? Metamorphic Fault Localization for Multi-Stage NLP Systems

*Amended per binding amendments BA-1 through BA-5 and recommended improvements R-1 through R-3.*

## Problem and Approach

When a multi-stage NLP pipeline — tokenizer → POS tagger → dependency parser → NER → downstream classifier — exhibits a behavioral inconsistency, every existing testing tool can detect it but none can identify which stage caused it. CheckList (Ribeiro et al., 2020) and TextFlint (Gui et al., 2021) apply meaning-preserving transformations and report end-to-end pass/fail. TextAttack generates adversarial inputs against single models. LangTest evaluates full pipelines holistically. The result: an NLP engineer who discovers that passivizing a sentence flips an entity label knows *that* the pipeline is broken, but not *whether* the fault originates in the tokenizer, the tagger, the parser, or the NER model — and not whether the parser merely amplified a tagger error or introduced a new one. This is the difference between a thermometer and an MRI. We build the MRI.

This project constructs a grammar-compiled metamorphic testing engine whose central contribution is **causal-differential fault localization across multi-stage NLP inference pipelines**. The engine operates in a closed loop: a probabilistic unification grammar generates linguistically-controlled test inputs (parameterized by voice, tense, embedding depth, agreement patterns); a composable algebra of meaning-preserving transformations applies systematic perturbations; per-stage pipeline instrumentation records intermediate representations; a causal-differential analysis identifies which stage introduced or amplified the inconsistency (not merely which stage correlates with failure); and a grammar-aware shrinker reduces fault-exposing inputs to minimal counterexamples — typically under 10 words — that a developer can immediately act on. The formal backbone is a transformation algebra (a free monoid quotiented by task-parameterized semantic equivalence) whose statistical composition theorem enables systematic coverage of the transformation space without exhaustive testing, and a Behavioral Fragility Index that quantifies how severely each pipeline stage amplifies perturbations.

**Primary targets** are classical pipelines (spaCy, Stanza) and transformer-based pipelines (HuggingFace), which together cover ~80% of production multi-stage NLP deployments. These pipelines have deterministic or near-deterministic inference, observable intermediate representations, and CPU-feasible execution — all prerequisites for the causal-differential analysis that is our central contribution. The engine produces both minimal counterexamples and a public behavioral atlas mapping (transformation × pipeline × task) triples to behavioral outcomes.

We are explicit about what this project is *not*. Metamorphic testing for NLP is a mature, crowded field — we build on it, we do not reinvent it. Grammar-based fuzzing for structured inputs is well-studied — we adapt it to natural language, we do not claim the technique is new. Individual meaning-preserving transformations (passivization, synonym substitution, clefting) are catalogued by TextFlint and NL-Augmenter. Our contribution is the *localization loop*: the ability to take any metamorphic violation, identify the causal pipeline stage, and hand the developer a minimal proof sentence. No existing tool provides this. The prior art audit (reviewing CheckList, TextFlint, METAL, LLMORPH, LangTest, TextAttack, Grammarinator, Nautilus, TreeReduce, and the full SBFL literature) confirms that pipeline-stage fault localization for NLP inference is a genuine gap with low novelty risk.

The output is not just a debugging tool but a **behavioral atlas**: a public database mapping (transformation × pipeline × NLP task) triples to behavioral outcomes across major frameworks. This atlas enables cross-system comparisons ("transformer-based NER is 3.2× more fragile to passivization than rule-based NER"), reveals systematic blind spots ("cleft constructions cause more NER failures than any other transformation across all tested pipelines"), and provides a citable scientific artifact for the NLP robustness community.

## Value Proposition

**Who needs this, and why desperately.** Teams running classical or transformer-based multi-stage NLP pipelines — particularly in regulated industries (healthcare NLP, legal NLP, financial NLP) — debug pipeline interactions by inserting print statements between stages. When a customer reports a misclassification, the engineer manually traces through tokenization, encoding, classification, and post-processing to find the fault. According to John Snow Labs' 2023 survey, 60% of NLP teams spend more time debugging pipeline interactions than training models. Cloud NLP API users (AWS Comprehend, Google Cloud NLP, Azure) cannot even inspect intermediate representations — they are stuck with black-box end-to-end testing. We acknowledge the industry's shift toward LLM-based systems; the primary audience for this tool is the substantial population of teams that still operate classical or transformer-based multi-stage pipelines, which remain dominant in regulated domains where model interpretability, reproducibility, and deterministic inference are requirements.

**What becomes possible.** An NLP engineer adds a configuration file specifying their pipeline topology and metamorphic relation suite. The engine runs, produces a structured report: "7 actionable inconsistencies found. Inconsistency #1: the POS tagger mishandles passivized gerunds, causing the dependency parser to misattach PPs, cascading to NER. Minimal proof: 'The report was being written by Kim.' Severity: BFI 4.7." The minimal counterexample becomes a permanent regression test. When a model checkpoint is swapped, the suite re-runs automatically. For regulated industries, this transforms "we tested it manually on 50 examples" into "we tested it on 14,000 grammar-valid metamorphic variants, localized 7 inconsistencies to specific pipeline stages, and reduced each to a minimal proof sentence."

**Operational definitions: "actionable inconsistency" vs. "expected behavior."** Not every behavioral difference under transformation is a bug. A dependency parser *should* produce different trees for active vs. passive voice — that is correct sensitivity, not a fault. We define an **actionable inconsistency** (colloquially, a "bug") as a violation where: (a) the metamorphic relation explicitly defines the behavioral dimension as invariant for the given NLP task (e.g., entity labels should be preserved under passivization for NER); (b) the violation exceeds a task-specific severity threshold (e.g., entity span mismatch, not mere confidence score fluctuation); and (c) the violation is reproducible across ≥3 distinct inputs exercising the same transformation. Conversely, **expected behavior** is a behavioral difference on a dimension that the MR does not constrain, or a violation below the severity threshold, or a non-reproducible one-off. This distinction is operationalized in the MR specification language, not left to post-hoc judgment.

**Why the full engine, not a 5K-line script.** A naive per-stage comparison script finds obvious faults but cannot: (a) distinguish "stage k introduced the fault" from "stage k amplified a fault from stage k-1" — this requires causal intervention, not mere observation; (b) systematically cover the transformation space — ad-hoc testing misses interaction effects that algebraic coverage exposes; (c) produce minimal counterexamples that respect grammaticality — string-level shrinking produces ungrammatical fragments that developers dismiss. The delta between naive and principled approaches is precisely the localization accuracy, coverage efficiency, and counterexample quality that make the tool trustworthy enough for production use.

## Technical Difficulty

Building this engine requires solving five hard subproblems, none of which has an off-the-shelf solution.

**1. Grammar Compiler with Dual Compilation (18K LoC, Rust).** A probabilistic unification grammar for English that enforces agreement, subcategorization, and selectional restrictions — features that context-free grammars cannot express. The compiler produces two automata from a single grammar specification: a weighted recursive transition network optimized for generation (forward sampling with Boltzmann distributions for naturalistic output) and a minimal-derivation lattice optimized for shrinking (backward search for smallest grammatical sub-derivations). The unification engine implements Tomabechi/Wroblewski DAG unification with occurs-check over re-entrant feature structures. The morphological engine handles English inflection, derivation, and irregular forms via finite-state transducers. Grammar naturalness is enforced through frequency-weighted lexical selection from corpus statistics and a perplexity proxy (trigram model, CPU-feasible) that constrains generation to sentences within 2× the perplexity of Penn Treebank text. This is not Grammarinator applied to English; linguistic grammars are massively ambiguous, context-sensitive, and require knowledge (binding theory, island constraints, control/raising) that ANTLR grammars do not capture.

**2. Transformation Algebra and Metamorphic Relations (12K LoC, Rust + Python).** Fifteen core linguistically-grounded transformations for the initial release — each 150-250 lines of tree-transduction logic encoding deep syntactic knowledge. The 15 transformations are: (1) passivization, (2) clefting, (3) topicalization, (4) relative clause insertion, (5) relative clause deletion, (6) tense change, (7) agreement perturbation, (8) synonym substitution, (9) negation insertion, (10) coordinated NP reordering, (11) PP attachment variation, (12) adverb repositioning, (13) there-insertion, (14) dative alternation, (15) embedding depth change. These 15 were selected to cover the major axes of syntactic variation (voice, information structure, clause structure, morphological agreement, lexical choice, constituent order, complexity) while keeping the initial release tractable. Each transformation explicitly documents its preservation scope per NLP task — we do not claim passivization preserves "meaning" in the abstract; we claim it preserves entity reference (for NER) and polarity (for sentiment) while changing dependency structure (for parsing). The algebra tracks composition properties (which pairs commute, which compositions preserve which semantic predicates) and the MR checker implements task-parameterized consistency predicates: entity-span equivalence for NER, polarity preservation for sentiment, τ-indexed structural equivalence for parsing. False positive rates are tracked and reported alongside bug counts.

**3. Causal-Differential Fault Localizer (12K LoC, Rust + Python).** The diamond contribution. For each metamorphic violation, the localizer computes per-stage differentials (divergence between original and transformed intermediate representations), applies spectrum-based fault localization (Ochiai coefficient) across a corpus of test outcomes, and then performs causal refinement: interventional analysis that replaces a suspected stage's output with the original execution's output and observes whether the downstream violation disappears. This distinguishes fault *introduction* from fault *amplification* — a distinction no existing NLP testing tool makes. We acknowledge the circularity concern: using the original execution as a reference assumes the transformation is meaning-preserving, which is the hypothesis under test. We frame the causal analysis as a *validated heuristic* — not a provably correct oracle — and measure its accuracy empirically against fault-injected ground truth, where the circularity does not apply because the injected fault's location is known.

**4. Grammar-Aware Shrinker (12K LoC, Rust).** Parse-tree delta debugging that maintains grammaticality at every step, preserves transformation applicability (a shrunk sentence must still be passivizable if the original was), and preserves the metamorphic violation. Operates on the shrinking automaton compiled from the grammar, pruning the exponential space of sub-derivations. Produces 1-minimal counterexamples (no single subtree replacement further reduces the input) with a convergence bound of O(|T|² · |R|) grammar-validity checks. Full minimality is NP-hard; 1-minimality is what practitioners need.

**5. Pipeline Instrumentor and Evaluation Infrastructure (15K LoC Python + Rust bridge, plus supporting subsystems).** Adapters for spaCy and HuggingFace Transformers (covering ~80% of production NLP pipelines), with a generic adapter for arbitrary Python pipelines. Intermediate representation capture with copy-on-write snapshots and delta compression for memory efficiency on laptops. Adaptive test scheduler with budget-constrained execution: given a CPU time budget (e.g., 30 minutes), multi-armed bandit allocation prioritizes high-value test inputs. Coverage analyzer tracks grammar × transformation × MR pairwise coverage. Counterexample database (SQLite-backed) with regression tracking and structured JSON/HTML reports.

The algorithmic core is the intellectual contribution. The infrastructure is not claimed as a contribution but is necessary for a self-contained, evaluable tool. The evaluation harness is necessary for the reproducible evaluation that validates the contribution.

The Rust+Python split is justified by profiling reality: grammar compilation, transformation composition, and shrinking are CPU-bound combinatorial operations where Rust's performance is essential; NLP pipeline execution is Python-native (spaCy, HuggingFace are Python libraries) so instrumentation must be Python. The PyO3 bridge (~3K LoC within the instrumentor) is the cost of this architecture. A pure-Python implementation would be ~35% smaller but would make grammar-compiled generation and grammar-aware shrinking impractically slow — the grammar engine must explore millions of derivations per second during shrinking, which Python cannot sustain.

**Minimum viable system (go/no-go milestone): ~50–60K LoC.** The minimum viable system that can demonstrate the core contribution consists of: the Fault Localizer (12K), Grammar-Aware Shrinker (12K), spaCy pipeline adapter (subset of Instrumentor, ~6K), 15 transformations with MR checker (~10K), Input Generator (~8K), and Tiers 1–2 evaluation harness (~5K). This ~53K LoC subset is sufficient to localize faults in spaCy pipelines, produce minimal counterexamples, and evaluate localization accuracy — the core claim. If this milestone is not achievable, the project does not proceed to the full ~107K build-out. The remaining ~54K (HuggingFace adapter, coverage analyzer, full spec language, test scheduler, behavioral atlas, Tiers 3–5 evaluation) extends the system from a proof-of-concept to a complete tool.

## New Mathematics Required

Two genuinely new mathematical contributions enable capabilities no existing tool provides. Two formal specifications provide practical metrics for the system. Three adapted results provide the formal foundation.

**M4: Causal-Differential Fault Localization (NEW, load-bearing).** Given pipeline P = s₁ → ... → sₙ and a metamorphic violation, compute per-stage differentials Δₖ(x, τ) = dist(sₖ(prefix_k(x)), sₖ(prefix_k(τ(x)))). Localize to k* = argmax_k [Δₖ - E[Δₖ | τ is meaning-preserving]]. The causal refinement distinguishes introduction from amplification via interventional analysis: replace stage k's output with the original execution's output and observe downstream behavior change. This is a novel synthesis of SBFL with interventional causal reasoning, instantiated for NLP pipeline stages where "coverage" means linguistic feature processing, not statement execution. Complexity: O(N · n · C_pipeline), linear in test count and stage count.

**M5: Grammar-Constrained Shrinking Convergence (NEW).** GCHDD (Grammar-Constrained Hierarchical Delta Debugging) terminates in O(|T|² · |R|) grammar-validity checks, each costing O(|F|) for feature unification. Produces 1-minimal counterexamples. This extends Zeller's delta debugging to operate on parse trees with grammaticality and transformation-applicability constraints — a synthesis not achieved by TreeReduce (which handles programming languages without linguistic feature unification) or Gordon's 2022 vision paper (which proposed the concept without implementation or convergence bounds).

**M3: MR Composition Theorem (FORMAL SPECIFICATION, enabling practical coverage optimization).** If transformations τ₁, τ₂ modify disjoint syntactic positions and satisfy task-parameterized meaning preservation independently, then their composition preserves meaning with probability ≥ 1-δ, where δ is empirically bounded via Clopper-Pearson exact binomial confidence intervals. We deliberately weaken this from a deterministic guarantee to a statistical one: natural language meaning is notoriously non-compositional (idioms, collocations, scope interactions violate syntactic locality), and a deterministic theorem would be false. The statistical form is honest and practically useful — it enables systematic coverage of the transformation space by testing individual transforms and inferring composition behavior, reducing the required test count from O(|T|²) to O(|T| · log|R|) for pairwise coverage. We do not claim this as a novel mathematical contribution; the statistical machinery (Clopper-Pearson intervals, union bounds) is standard. The contribution is the formal specification that connects transformation composition to coverage optimization in the NLP metamorphic testing setting.

**M7: Behavioral Fragility Index (FORMAL SPECIFICATION, providing an interpretable per-stage metric).** BFI(P, τ) = E_x[dist_out(P(x), P(τ(x)))] / E_x[dist_in(x, τ(x))], estimated empirically from the test corpus. BFI >> 1 indicates the pipeline amplifies the transformation's effect; BFI ≈ 1 indicates stability. We do *not* rely on a Lipschitz composition bound for practical computation — NLP pipeline stages are discrete and often discontinuous, violating Lipschitz continuity. The theoretical bound (∏Lₖ · dist_in) serves only as motivation for why amplification happens; the BFI itself is a purely empirical metric with task-specific distance functions (tree edit distance for parse trees, span-level F1 for NER, score difference for sentiment). We acknowledge that BFI >> 1 can reflect *correct* sensitivity (a parser *should* produce different trees for active vs. passive voice) — the metric is interpretable only relative to the task-specific MR that defines which behavioral dimensions should be invariant. We do not claim the ratio metric itself as novel mathematics; the contribution is the formal specification that makes per-stage fragility comparable across pipelines and transformations.

**Summary:** Two genuinely new mathematical contributions (M4, M5) and two formal specifications (M3, M7) that enable practical use. M4 is the load-bearing novelty; M5 solves a new constraint optimization problem; M3 and M7 formalize metrics that make the system usable and evaluable.

**M1 (ADAPT): Transformation Algebra** — free monoid Σ*/≡_s with task-parameterized semantic equivalence. Standard algebraic structure; the contribution is the specific equivalence relation grounded in NLP task semantics and the computed commutativity relation that enables coverage optimization. **M2 (ADAPT): Probabilistic Unification Grammar** — well-studied formalism (PATR-II, LFG); the contribution is dual compilation to generation + shrinking automata from a single specification. **M6 (ADAPT): Product-Space Coverage** — covering arrays (Colbourn 2004) over grammar × transformation × MR; standard technique applied to a new product space for stopping criteria and test prioritization.

## Best Paper Argument

This work would be a strong best-paper candidate at ISSTA or ASE because it solves the NLP pipeline debugging problem with a formally grounded approach that demonstrably finds real bugs, localizes them to specific stages, and produces minimal proofs — capabilities no existing tool provides. We are honest: strong accept potential, with best-paper depending on the quality of real bug discoveries.

**The qualitative shift.** Before this work: "the pipeline is inconsistent on passivized inputs." After this work: "the POS tagger mishandles passivized gerunds, causing the dependency parser to misattach PPs, cascading to NER — here is the 5-word sentence that proves it." This is a change in *kind*, not degree. CheckList told us testing matters; we show how to make testing *diagnostic*.

**The centerpiece artifact.** A table showing 10 real bugs in shipped NLP systems, each with a minimal counterexample of ≤10 words, each localized to a specific pipeline stage — that table is the kind of result reviewers remember, cite, and put in their talks. This "10 bugs, 10 words" artifact is the paper's most memorable deliverable: compact, verifiable, and immediately useful as a benchmark. The behavioral atlas — a public database of behavioral outcomes across transformations, pipelines, and tasks — becomes a community resource that other researchers build on.

**The predictability defense.** A reviewer may object that combining SBFL with interventional analysis on NLP pipeline stages is a "natural" or "obvious" idea. We acknowledge this: the high-level approach is indeed natural, and that is a feature, not a bug. The contribution is that *naive application fails* — vanilla SBFL without causal refinement conflates fault introduction with fault amplification, producing systematically wrong localizations on cascading faults (our Tier 2 evaluation quantifies this). The refinement that makes it work (interventional causal analysis with task-specific intermediate representation distances) is where the technical depth lies. Obvious-sounding ideas that require non-trivial engineering to realize are precisely what the SE testing community values.

**GPT-4-as-debugger baseline.** We include a baseline where GPT-4 is given the same pipeline topology, transformation, and end-to-end violation, and asked to localize the fault. This baseline is important because it represents the realistic alternative: an engineer using an LLM to debug their pipeline. If our tool outperforms GPT-4 on localization accuracy — particularly on cascading faults where the causal chain is non-obvious — that is a compelling demonstration of the tool's value over the strongest informal alternative.

**Why ISSTA/ASE over ACL/EMNLP.** The SE testing community values formal grounding (transformation algebra, convergence bounds, causal localization) and tool contributions (finding real bugs in real systems). At an NLP venue, reviewers may dismiss this as "CheckList + SBFL"; at a testing venue, the NLP domain instantiation is novel, the causal-differential localization is a genuine algorithmic contribution, and the grammar-aware shrinker solves a new constraint optimization problem. Secondary submission to ACL/EMNLP is viable if positioned as an NLP debugging contribution — not a testing contribution — with the behavioral atlas as a scientific deliverable.

## Evaluation Plan

All evaluation is fully automated with zero human annotation or human studies. We acknowledge that external validity of meaning-preservation and input naturalness cannot be perfectly assessed without human judgment — this is an explicit limitation. We validate internal metrics rigorously and use proxy metrics for naturalness.

**Operational definition of "actionable inconsistency."** A reported violation is an actionable inconsistency (i.e., a real bug worth reporting) if and only if: (a) the metamorphic relation explicitly defines the violated behavioral dimension as invariant for the given NLP task; (b) the violation exceeds a task-specific severity threshold (e.g., entity span mismatch for NER, polarity flip for sentiment — not mere confidence score fluctuation within ε); and (c) the violation is reproducible across ≥3 distinct inputs exercising the same transformation. Violations that fail any of these three criteria are classified as expected behavior or noise and are not counted in bug totals. This operationalization is encoded in the MR specification language and applied automatically.

**Tier 1: Real Bug Discovery.** Run the engine on three production-grade pipelines: spaCy `en_core_web_sm` and `en_core_web_trf`, HuggingFace `dslim/bert-base-NER` + `textattack/bert-base-uncased-SST-2`, and Stanza's full English pipeline. Target: ≥25 previously unknown, reproducible actionable inconsistencies (not injected faults — bugs that already exist in shipped systems), with ≥5 per pipeline. File reproducible reports on framework GitHub repos. If any are acknowledged or fixed by maintainers, that is gold-standard validation.

**Tier 2: Localization Accuracy.** Create 50 fault-injected pipeline configurations with known ground-truth faulty stages (tokenizer corruption, POS label swaps, dependency arc perturbation, NER label noise). Measure top-1 localization accuracy (target: ≥85%), top-2 accuracy (target: ≥95%). Compare against baselines: random stage selection (~20% for 5-stage pipeline), last-stage heuristic, vanilla SBFL without causal refinement, and established SBFL variants (Ochiai, DStar, Barinel) to ensure we compare against the strongest known spectrum-based techniques, not just naive baselines. Target: ≥20 percentage-point improvement over the best SBFL variant without causal refinement, with the largest gains on cascading faults where causal refinement distinguishes introduction from amplification. We acknowledge that injected faults may not represent real fault patterns (injections are uniform; real faults cluster around specific linguistic phenomena). We partially bridge this gap by cross-validating: for every real bug discovered in Tier 1, we check whether the localizer's stage-level diagnosis is consistent with manual root-cause analysis of the pipeline's intermediate representations, providing an external-validity estimate for the fault-injection results.

**Tier 3: Shrinking Quality.** Measure shrink ratio (target: ≥5× reduction, e.g., 40 words → 8 words average), shrinking time (target: <60 seconds per counterexample on CPU), and grammaticality preservation (100% of shrunk examples must parse correctly). Naturalness proxy: generated and shrunk sentences must have trigram perplexity within 2× of Penn Treebank sentences, measured by a CPU-feasible n-gram model.

**Tier 4: Coverage Efficiency.** Demonstrate that the statistical composition theorem enables ≥90% pairwise (grammar production × transformation) coverage with ≤5,000 test cases, compared to ≥50,000 test cases for both uniform random sampling and stratified sampling (stratified by transformation type) to achieve the same coverage — a ≥10× efficiency gain over the strongest sampling baseline.

**Tier 5: Behavioral Atlas.** Release a public database mapping (transformation × pipeline × task) → behavioral outcome for all tested pipelines. Report at least 3 cross-system findings (e.g., "transformer-based NER is N× more fragile to passivization than rule-based NER"). The atlas is a released artifact, not just reported numbers.

**False positive rate.** For every reported inconsistency, verify that the triggering transformation is meaning-preserving for the tested NLP task by checking against the task-parameterized MR specification. Report the false positive rate (target: <10%). This is automatable because the MR specifications define preservation scope explicitly — the check is "does passivization preserve entity reference?" not "does passivization preserve meaning in general."

**Negative control.** If fewer than 5 actionable bugs are found in Tier 1 (indicating that shipped pipelines may be more robust than expected, or that our transformations do not exercise real fault patterns), the paper pivots to evaluating the engine's standalone contributions: localization accuracy on injected faults (Tier 2), shrinking quality (Tier 3), and coverage efficiency (Tier 4). These tiers are independently publishable as tool contributions even if the real-bug yield is low. We state this contingency explicitly to avoid a paper whose central claim depends on an empirical outcome we cannot guarantee.

## Laptop CPU Feasibility

The system's CPU feasibility varies by pipeline class. We provide honest per-class estimates.

**Generation is fast.** Grammar-compiled generation from the Rust engine produces thousands of sentences per second on CPU. Grammar compilation is a one-time cost. The generation bottleneck is not computation but linguistic coverage.

**Statistical pipelines (~75 minutes, interactive debugging).** spaCy `en_core_web_sm` and Stanza statistical models run at ~2ms/sentence. Each test case requires two pipeline runs (original + transformed), so ~4ms per test case. The composition theorem reduces the required test count: ≥90% pairwise coverage requires ~5,000 test cases, not 50,000. At 4ms each: 5,000 × 0.004s ≈ 20 seconds for test execution. The real time is in localization (6 pipeline runs per violation, ~100 violations: ~2 minutes) and shrinking (top-20 violations, ~2,000 pipeline runs each at <60s per counterexample: ~20 minutes). **Total: ~75 minutes for a full test-localize-shrink cycle on a laptop CPU.** This is fast enough for interactive debugging during development.

**Transformer pipelines (~3–4 hours, nightly CI).** spaCy `en_core_web_trf` and HuggingFace BERT-based models run at ~200-500ms/sentence on CPU. At 500ms/inference, 5,000 test cases take ~80 minutes. Shrinking takes longer (~5 minutes per counterexample). **A full cycle takes ~3–4 hours.** This is a CI nightly-run, not an interactive debugging session — and is explicitly documented as such. The adaptive scheduler uses multi-armed bandit allocation to prioritize high-value tests within a user-specified CPU budget (e.g., "test for 30 minutes"), maximizing bug discovery per CPU-hour.

**RAG/LLM pipelines: NOT CPU-feasible.** RAG and LLM-based pipelines require either GPU inference or API calls with associated latency and cost. A single RAG pipeline invocation (retrieval + reranking + LLM generation) takes 1–10 seconds even with GPU; on CPU, LLM inference alone can take minutes per sentence. A 5,000-test-case cycle would take days on CPU. Additionally, LLM non-determinism (temperature > 0, sampling variability) violates the reproducibility assumption underlying our causal-differential analysis. **RAG/LLM pipeline support is out of the core scope** and is discussed in Future Extensions.

**Budget-constrained execution.** The composition theorem reduces the required test count from O(|T|²) to O(|T| · log|R|) for pairwise coverage, enabling ≥90% coverage with ~5,000 test cases instead of ~50,000 — a 10× reduction that is the difference between feasible and infeasible on CPU. The adaptive scheduler further optimizes within any user-specified budget.

**No per-run human involvement.** Generation is automated (grammar-compiled). Transformation is automated (tree-transduction). Oracle checking is automated (task-parameterized MR predicates). Localization is automated (causal-differential analysis). Shrinking is automated (grammar-constrained delta debugging). Evaluation uses automated metrics (localization accuracy on injected faults, shrink ratio, perplexity proxy for naturalness, coverage metrics). However, we are honest about amortized human costs: the grammar specification, transformation implementations, and metamorphic relation definitions require expert linguistic and NLP engineering as a one-time authoring cost. These are provided as part of the tool; users do not write them. The only human-requiring validation we cannot automate — meaning-preservation and naturalness judgment — is documented as an explicit limitation, with proxy metrics (perplexity, false positive rate) as honest substitutes.

## Future Extensions

**RAG/agent pipeline support.** The causal-differential fault localization framework is architecturally compatible with RAG pipelines (retriever → reranker → prompt constructor → LLM → output parser), which are inherently multi-stage systems with the same localization problem. However, three open problems must be solved before this extension is viable: (a) **non-deterministic LLM stage handling** — LLM inference with temperature > 0 produces variable outputs, requiring either deterministic decoding constraints or statistical aggregation over multiple runs to establish reliable baselines for causal-differential analysis; (b) **heterogeneous intermediate representation distance metrics** — RAG stages produce fundamentally different representations (document rankings, prompt strings, token probability distributions) requiring domain-specific distance functions that do not yet exist in a unified framework; (c) **GPU or API-based inference** — LLM stages are not CPU-feasible, requiring either local GPU or API budget, which changes the cost model entirely. We estimate RAG pipeline support as a follow-up paper building on the infrastructure established here.

**Multilingual support.** The grammar specification and all 15 transformations are English-specific. The path to multilingual support is via parameterized grammar specifications: the grammar compiler and transformation algebra are language-agnostic, but each target language requires its own grammar, morphological engine, and transformation implementations. This is a substantial but well-scoped engineering effort per language.

**Additional transformations.** The 15 core transformations cover the major axes of syntactic variation for English. The transformation algebra is designed to be extensible; additional transformations (e.g., extraposition, heavy NP shift, gapping, sluicing, pseudocleft) can be added incrementally without modifying the core engine. We estimate the full English transformation suite at ~30–40 transformations, to be developed based on empirical evidence of which transformations expose the most bugs in Tier 1 evaluation.
