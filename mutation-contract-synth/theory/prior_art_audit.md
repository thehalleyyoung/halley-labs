# Prior Art Audit: Mutation-Driven Contract Synthesis Engine

**Auditor Role**: Prior Art Auditor (Crystallization Team)
**Community**: Programming Languages and Formal Methods
**Date**: 2025-07-17

## Seed Idea Under Audit

> An engine that synthesizes formally verified function contracts by analyzing
> which code mutations kill existing tests, using the mutation–survival boundary
> to infer minimal pre/postconditions via SyGuS and verifying them against source
> via bounded SMT to catch latent bugs.

**Core pipeline**: Mutation analysis → killed/survived boundary extraction →
SyGuS-based contract synthesis → bounded SMT verification → latent bug detection.

---

## 1. MUTATION TESTING LITERATURE

### 1a. PIT, Major, Stryker, µJava

These are the standard mutation testing frameworks (PIT for Java/JVM, Major for
Java, Stryker for JavaScript/C#/.NET, µJava for Java). **None of these tools
infer specifications or contracts from mutation results.** They serve a single
purpose: measuring test suite adequacy by computing mutation scores
(killed/total non-equivalent mutants).

- **Closeness**: 2/10. These provide the raw mutation infrastructure the seed
  idea would build upon, but they never cross the bridge from mutation results
  to specification inference.
- **Delta**: Fundamental. These tools output a score; the seed idea uses
  mutation results as input signal to a synthesis pipeline.

### 1b. Mutation-Based Test Generation (µTest and related)

µTest and similar tools (e.g., MuJava-based test generators) use mutation
information to *generate tests that kill mutants*. The direction is
mutation → tests, not mutation → specifications.

- **Closeness**: 3/10. Shares the insight that mutation results carry
  semantic information, but targets test generation, not contract synthesis.
- **Delta**: Fundamental difference in output artifact (tests vs. contracts).

### 1c. Mutation Score as Specification Adequacy Metric

**Key paper**: Knüppel, Schulze, Schaefer. *"How much Specification is Enough?
Mutation Analysis for Software Contracts"* (FormaliSE 2021).

This paper uses mutation analysis to **evaluate** the adequacy of existing
contracts (JML annotations checked with KeY). It asks: "Are these specs strong
enough to catch mutants?" It does NOT synthesize new specifications from
mutation results.

- **Closeness**: 5/10. Establishes the theoretical connection between mutation
  analysis and specification quality. Uses the same conceptual bridge
  (mutation ↔ specification) but in the evaluation direction only.
- **Delta**: The seed idea inverts this: instead of evaluating specs via
  mutation, it *synthesizes* specs from mutation signals. This is a
  fundamental directional reversal.

### 1d. IronSpec (OSDI 2024) and MutDafny

IronSpec mutates *specifications* (not code) to find specification bugs in
Dafny programs. MutDafny is a related tool from U. Porto. Both assess
whether formal specs are adequate by testing them against mutations.

- **Closeness**: 4/10. Operates in the specification ↔ mutation space but in
  the opposite direction: mutates specs to test them, rather than using code
  mutation results to synthesize specs.
- **Delta**: Fundamental directional difference. IronSpec tests existing
  specs; the seed idea creates specs from scratch.

## 2. CONTRACT/SPECIFICATION INFERENCE

### 2a. Daikon (Dynamic Invariant Detection)

Daikon (Ernst et al., ICSE 1999+) observes program executions to discover
"likely invariants" — properties that hold across all observed traces. It
uses template matching over runtime values at program points.

- **Closeness**: 4/10. Both Daikon and the seed idea aim to infer
  specifications automatically. But the signal sources are completely
  different: Daikon uses observed runtime values; the seed idea uses the
  killed/survived mutation boundary.
- **Delta**: Fundamental difference in signal source (observation vs.
  fault-injection). Daikon produces likely (unverified) invariants; the seed
  idea produces SMT-verified contracts. Daikon's templates are fixed; the
  seed idea uses SyGuS for open-ended synthesis.

### 2b. SpecFuzzer (ICSE 2022, Molina et al.) ⚠️ CLOSEST PRIOR ART

SpecFuzzer combines three techniques:
1. Grammar-based fuzzing to generate candidate assertions
2. Dynamic invariant detection (Daikon) to filter candidates
3. **Mutation analysis** to cluster and rank surviving assertions

SpecFuzzer is the closest prior art because it explicitly uses mutation
analysis in a specification inference pipeline.

- **Closeness**: 7/10. This is the most threatening overlap.
- **Key differences**:
  - **Signal role**: In SpecFuzzer, mutation analysis is used as a
    *filtering/ranking* step on candidates generated by fuzzing + Daikon.
    In the seed idea, mutation is the *primary synthesis signal* — the
    killed/survived boundary directly drives what contracts to synthesize.
  - **Synthesis engine**: SpecFuzzer uses grammar-based fuzzing (enumerate +
    filter). The seed idea uses SyGuS (counterexample-guided inductive
    synthesis) — a fundamentally different synthesis paradigm.
  - **Verification**: SpecFuzzer produces dynamically-validated assertions
    (no formal guarantee). The seed idea verifies via bounded SMT.
  - **Contract expressiveness**: SpecFuzzer produces JML-like assertions
    within a fixed grammar. SyGuS can synthesize over richer theory-specific
    grammars.
- **Delta**: Incremental in concept (both link mutation to specs), but
  **fundamental in mechanism** (fuzzing+filter vs. boundary-driven SyGuS
  synthesis, unverified vs. SMT-verified).

### 2c. EvoSpex (ICSE 2021, Molina & Aguirre)

EvoSpex uses evolutionary algorithms (genetic algorithms) to learn method
postconditions. It mutates execution traces to create negative examples, then
evolves assertions that separate valid from invalid states.

- **Closeness**: 5/10. Shares the insight of using "negative" (mutated)
  examples to drive specification learning.
- **Delta**: Uses evolutionary search (GA), not SyGuS. Mutates traces, not
  code. No formal verification. Postconditions only (no preconditions).

### 2d. CodeContracts (Microsoft)

Static inference of .NET contracts using abstract interpretation. Infers
non-null constraints, bounds, and simple relational properties.

- **Closeness**: 3/10. Both infer contracts, but CodeContracts uses
  abstract interpretation (no mutation, no synthesis, no tests).
- **Delta**: Completely different methodology (static AI vs. dynamic
  mutation-driven synthesis).

### 2e. GAssert

Dynamic invariant detection tool that generates assertions from execution
traces. Predecessor to SpecFuzzer.

- **Closeness**: 3/10. Pure dynamic analysis, no mutation component.
- **Delta**: Fundamental (trace-based observation vs. mutation boundary).

### 2f. Houdini (Flanagan & Leino, FME 2001)

Modular annotation inference via guess-and-refute: generates candidate
annotations, uses static verification to iteratively refute false ones.

- **Closeness**: 4/10. Shares the "generate candidates, verify, refine"
  loop structure. But the candidate generation uses static templates (not
  mutation signals) and refutation uses a static verifier (not SyGuS/SMT
  on mutation data).
- **Delta**: Different signal source (static templates vs. mutation
  boundary), different synthesis strategy.

### 2g. Abstract Interpretation (Astrée, Infer)

Astrée and Facebook Infer infer program properties via abstract
interpretation (sound over-approximation). They detect potential errors
without mutation or test-driven signals.

- **Closeness**: 2/10. Different methodology entirely (static AI vs.
  dynamic mutation-driven synthesis).
- **Delta**: Fundamental.

## 3. SyGuS AND SYNTHESIS

### 3a. SyGuS Competition Solvers (CVC5, EUSolver)

These are general-purpose SyGuS solvers. They can synthesize functions
meeting semantic constraints within syntactic grammars. The seed idea
would use these as the synthesis backend.

- **Closeness**: 3/10. These are tools, not approaches. SyGuS has been
  used for loop invariant synthesis (e.g., LoopInvGen) but not directly
  driven by mutation analysis signals.
- **Delta**: The seed idea's novelty is not in SyGuS itself but in the
  signal source (mutation boundary) feeding the SyGuS problem encoding.

### 3b. SyGuS for Specification Inference

SyGuS has been used for loop invariant synthesis (LoopInvGen, ICE-DT) and
for precondition inference in verification settings. These are driven by
program semantics and counterexamples, not by mutation testing data.

- **Closeness**: 5/10. The synthesis mechanism is the same, but the input
  signal is different.
- **Delta**: Incremental on the synthesis side; fundamental on the signal
  source side. The key contribution is *what feeds into* SyGuS, not
  SyGuS itself.

### 3c. Sketch-Based Synthesis (Solar-Lezama)

Sketch synthesizes implementations from partial programs using CEGIS.
It targets implementation synthesis (filling holes in code), not
specification inference.

- **Closeness**: 3/10. Shares CEGIS loop structure with SyGuS-based
  approaches but targets a different artifact (implementations, not specs).
- **Delta**: Different synthesis target.

## 4. FORMAL VERIFICATION OF INFERRED SPECS

### 4a. Bounded Model Checking (CBMC, JBMC)

These verify programs against assertions/contracts within bounded
executions. They could serve as the verification backend for the seed
idea. JBMC has been used to verify JML contracts.

- **Closeness**: 3/10. Verification tools, not inference tools.
- **Delta**: The seed idea uses BMC as one component; the novelty is in
  what BMC receives (mutation-synthesized contracts).

### 4b. Dafny, F*, Liquid Haskell

These are languages with integrated specification and verification. They
verify human-written or LLM-generated specs. None use mutation analysis
as a signal for spec inference.

- **Closeness**: 2/10. Verification ecosystems, not inference tools.
- **Delta**: Fundamental (verification of existing specs vs. synthesis
  of new specs from mutation signals).

### 4c. Auto-Active Verification (Why3, KeY)

Why3 and KeY verify annotated programs interactively/semi-automatically.
KeY was used in the FormaliSE 2021 work to verify JML contracts for
mutation adequacy assessment, but not for synthesis.

- **Closeness**: 3/10. Could serve as verification backend.
- **Delta**: These are verifiers, not synthesizers.

## 5. COMBINED APPROACHES (MOST CRITICAL)

### 5a. Has ANYONE combined mutation analysis with specification synthesis?

**Yes, partially.** The landscape:

| Work | Mutation Role | Synthesis Method | Verification | Distance |
|------|--------------|-----------------|-------------|----------|
| **SpecFuzzer** | Filter/rank candidates | Grammar fuzzing + Daikon | Dynamic only | **7/10** |
| **EvoSpex** | Negative examples from trace mutation | Genetic algorithm | Dynamic only | 5/10 |
| **FormaliSE 2021** | Evaluate existing specs | None (evaluation only) | KeY verifier | 5/10 |
| **AutoSpec** | Evaluation metric | LLM + iterative refinement | Frama-C | 4/10 |
| **ClassInvGen** | Evaluation metric | LLM co-generation | Test-based | 3/10 |
| **Beyond Postconditions** | Evaluation metric | LLM NL→Contract | Verifier feedback | 3/10 |
| **Gambit/Certora** | Spec evaluation in smart contracts | Manual specs | Formal prover | 3/10 |
| **Seed Idea** | **Primary synthesis signal** | **SyGuS** | **Bounded SMT** | — |

### 5b. SpecMutator / MutSpec / Mutation↔Specification Duality

No tool or paper with exactly these names exists. The closest conceptual
framework is the FormaliSE 2021 paper's "mutation analysis for contracts"
and IronSpec's "specification mutation testing."

The **mutation↔specification duality** — the idea that the set of mutants
killed by a test suite implicitly defines a specification, and conversely
that a specification defines which mutants it can catch — is discussed
informally in several papers but has NOT been formalized into a synthesis
algorithm. The seed idea is the first to propose making this duality
*constructive* (actually synthesizing the specification from the boundary).

### 5c. Test-Driven Specification Inference

Several tools infer specs from test behavior (Daikon from traces, EvoSpex
from test-generated states, SpecFuzzer using tests as oracle). The seed
idea differs by using *mutation test results* (a derived signal from tests)
rather than raw test execution data.

### 5d. POPL/PLDI/OOPSLA Papers Connecting Mutations to Contracts (2015-2024)

After systematic search:
- **No POPL/PLDI/OOPSLA paper** in this period directly proposes synthesizing
  contracts from mutation analysis.
- The closest top-tier work is:
  - Synthesis of loop invariants via CEGIS (POPL/PLDI, multiple years) —
    different signal source
  - SpecFuzzer at ICSE 2022 — not a top PL venue
  - IronSpec at OSDI 2024 — spec testing, not synthesis
  - AutoSpec at CAV 2024 — LLM-based, mutation as metric only

**This is a significant finding**: the specific combination of mutation
analysis → specification synthesis has NOT appeared at top PL venues.

## 6. PORTFOLIO OVERLAP CHECK

### 6a. cross-lang-verifier (SemRec)

- **What**: C↔Rust semantic equivalence verification at source level
- **Formalism**: SMT-based bitvector constraints for overflow semantics
- **Overlap with seed idea**: **Minimal.** Different problem (equivalence
  verification vs. contract synthesis), different input (two codebases vs.
  one codebase + tests), different output (equivalence verdict vs. contracts).
  Shares SMT as backend technology only.

### 6b. synbio-verifier (BioProver)

- **What**: Formal verification + repair of synthetic biology gene circuits
- **Formalism**: δ-decidable BMC + moment closure + CEGAR
- **Overlap with seed idea**: **None.** Completely different domain
  (biology vs. software), different formalism, different artifacts.

### 6c. dp-verify-repair (DP-CEGAR)

- **What**: Differential privacy verification and repair for Python
- **Formalism**: Symbolic execution + SMT counterexample search
- **Overlap with seed idea**: **Minimal.** Different property (privacy vs.
  functional correctness), different signal source (symbolic execution vs.
  mutation analysis). Shares SMT as backend.

### 6d. tensorguard

- **What**: Tensor shape correctness verification for ML pipelines
- **Formalism**: Type/shape checking
- **Overlap with seed idea**: **None.** Different domain (ML shapes vs.
  program contracts), different methodology entirely.

**Portfolio verdict**: No significant overlap with any portfolio project.
The shared use of SMT technology is at the infrastructure level, not the
conceptual level.

## 7. NOVELTY VERDICT

### Overall Novelty Rating: **7/10**

### Justification

The seed idea sits at an **unexploited intersection** of three mature areas:
mutation testing, SyGuS-based synthesis, and bounded SMT verification. Each
component is well-established, but their specific combination — and
particularly the use of the mutation-survival boundary as the *primary
constructive signal* for specification synthesis — has not been demonstrated.

**What prevents a higher rating (8-10):**
- SpecFuzzer (ICSE 2022) already links mutation analysis to specification
  inference, even if the mechanism differs significantly.
- The individual components (mutation testing, SyGuS, SMT) are all mature
  and well-understood. The novelty is in their combination and the specific
  information-flow architecture, not in new theoretical primitives.
- LLM-based approaches (AutoSpec, ClassInvGen, Beyond Postconditions) are
  rapidly advancing and may reach similar capabilities through different
  means, reducing the practical significance window.

**What prevents a lower rating (1-5):**
- No existing system uses the killed/survived mutation boundary as the
  *primary input signal* to a synthesis engine for contracts. This is a
  genuine conceptual contribution.
- The SyGuS encoding of mutation-derived constraints is novel and non-trivial.
- The end-to-end pipeline (mutation → boundary → SyGuS → SMT verification →
  bug detection) does not exist in any published system.
- The absence of this specific combination at top PL venues (POPL/PLDI/
  OOPSLA) over the past decade is strong evidence of genuine novelty.

### STRONGEST Novelty Claim

**"The mutation-survival boundary as a constructive specification signal."**

No prior work treats the killed/survived partition as a *specification* to be
formalized. Existing work uses mutation scores as *metrics* (FormaliSE 2021),
as *filters* (SpecFuzzer), or as *negative examples* (EvoSpex). The seed idea
uniquely treats the boundary itself as an implicit contract definition that
can be made explicit through synthesis. This makes the
mutation↔specification duality *constructive* rather than merely analytical.

### WEAKEST Novelty Claim

**"Using SyGuS for contract synthesis"** and **"using bounded SMT for
verification of inferred contracts."**

Both SyGuS-based specification inference (LoopInvGen, ICE-DT for invariants)
and bounded SMT verification of contracts (CBMC/JBMC with JML) are
established techniques. The novelty of these components comes only from
their integration with the mutation signal, not from their independent use.
If the mutation-boundary-as-signal contribution is not sufficiently
differentiated from SpecFuzzer's use of mutation for filtering, the
remaining pipeline is largely a composition of known tools.

### Risk Assessment

| Risk | Level | Mitigation |
|------|-------|-----------|
| SpecFuzzer overlap | **Medium-High** | Emphasize: boundary-as-signal vs. filter, SyGuS vs. fuzzing, verified vs. unverified output |
| LLM-based approaches overtaking | **Medium** | Position as principled/formal alternative to LLM-based approaches; emphasize guarantees |
| "Just engineering" critique | **Medium** | Need strong theoretical contribution (e.g., formalize mutation↔spec duality, prove minimality guarantees) |
| Scalability concerns | **Medium** | Mutation testing is expensive; SyGuS has scalability limits. Need convincing evaluation |

### Recommendation

**Proceed with the idea**, but with these critical adjustments:

1. **Explicitly formalize the mutation↔specification duality** as a
   theoretical contribution. Define precisely when the killed/survived
   boundary uniquely determines a minimal contract.

2. **Position carefully against SpecFuzzer.** The distinction is real
   (signal source vs. filter, synthesis paradigm, verification guarantee)
   but must be argued precisely.

3. **Emphasize the formal verification loop.** The fact that output
   contracts are SMT-verified (not just dynamically validated) is a
   genuine differentiator from ALL prior spec inference work except
   Houdini-style static approaches.

4. **Demonstrate latent bug detection** — cases where the mutation-inferred
   contracts catch real bugs that existing tests miss. This is the killer
   application that no prior work delivers end-to-end.
