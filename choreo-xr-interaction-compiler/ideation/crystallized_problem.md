# Choreo: A Compiler for Spatial-Temporal Interaction Choreographies with Decidable Verification

## Problem Statement

Mixed-reality applications encode intricate multi-party interaction protocols—a user reaches toward a holographic panel, a virtual agent responds with a gesture sequence, a spatial anchor constrains a shared annotation to a tabletop region, and a timeout resets the scene if no gaze fixation is detected within two seconds—yet the field has no formal language, no compiler, and no verification infrastructure for these choreographies. Every XR framework (Unity MRTK, Meta Interaction SDK, Apple RealityKit, WebXR) re-implements interaction logic as imperative spaghetti: ad-hoc boolean flags, scattered coroutines, and engine-coupled callbacks that are platform-locked, untestable without a headset, and unanalyzable by any tool. There is no `pytest` for XR interaction logic. CI pipelines cannot cover spatial-temporal behavior. Regressions ship silently, and deadlocks surface only in live demos.

The core insight is that the *interaction-relevant* subset of a mixed-reality scene can be formalized as a network of **spatial-temporal event automata**: finite-state machines whose transitions are guarded by spatial predicates (containment, proximity, gaze-cone intersection) evaluated over an R-tree index, and whose timing constraints are expressed in a bounded fragment of Metric Temporal Logic. This formalization is the missing bridge between the continuous 3D world and the discrete world of formal verification. Critically, not all Boolean valuations of spatial predicates are geometrically realizable—monotonicity, the triangle inequality, and containment consistency constrain the feasible predicate space, and exploiting this structure is the key to making verification tractable.

**Choreo** is a domain-specific language and compiler that brings programming-languages methodology—formal semantics, static type checking, compilation, and verification—to XR interaction for the first time. Developers declare interaction choreographies in a spatial-temporal DSL: `region panel = box(table, 0.3m); interaction menu_open = gaze(panel) & reach(panel, <500ms) → activate(menu)`. The compiler performs spatial type checking (are referenced zones geometrically consistent?), temporal well-formedness analysis (are timeout constraints satisfiable?), and determinism checking (can two patterns fire simultaneously?), then lowers the specification through an Event Calculus intermediate representation into a network of R-tree-backed event automata. These compiled automata serve dual purposes: they execute headlessly on CPU for testing and simulation, and they are directly amenable to BDD- and SAT-based reachability and deadlock verification with geometric consistency pruning to tame the state-space explosion.

The field has standardized content interchange (glTF, USD), rendering APIs (Vulkan, WebGPU), and tracking interfaces (OpenXR), but has conspicuously failed to standardize *interaction*. Choreo fills this gap. The analogy is to Halide for image processing, Cg for GPU shading, and TVM for ML compilation: domains that were programming at the wrong abstraction level until a DSL and compiler revealed the right one. A working Choreo compiler that finds real deadlocks in production MRTK interaction graphs—bugs that shipped to users—would demonstrate that XR interaction has been programming at the wrong abstraction level, and that the programming-languages approach yields immediate practical dividends.

## Value Proposition

**XR platform teams** (Meta, Apple, Microsoft, Qualcomm) ship interaction toolkits with hundreds of cross-cutting interaction rules. Every release risks regressions that surface only during manual QA playtest weeks. A headless verifier running in CI would collapse weeks of testing into minutes of automated checking. **Accessibility engineers** need to prove that alternative input modalities (eye-tracking, switch scanning, voice) reach the same set of scene states as hand tracking—a reachability question that a model checker answers exhaustively, whereas manual testing covers a fraction of paths. **Multi-user XR developers** (Spatial, Microsoft Mesh) face combinatorial explosion of interaction interleavings between co-located users that no manual test plan can cover. **Standards bodies** (OpenXR, Khronos, W3C WebXR) working toward XR interoperability have no way to specify interaction *behavior* portably: USD describes scenes, glTF describes assets, but nothing describes how a user interacts with them. **Research labs** publishing interaction techniques at CHI and UIST currently share videos and pseudocode; a Choreo program would be a reproducible, executable, verifiable artifact.

What becomes possible that was not before: (1) CI/CD for spatial-temporal interaction behavior—the first headless correctness check for XR choreographies; (2) deadlock and unreachable-state detection across interaction patterns, catching bugs that currently require live demos to discover; (3) portable interaction specifications that compile to identical event-level behavior on different runtimes; (4) **exhaustive accessibility audits** verifying that every scene state is reachable via every supported input modality—the first tool that can *prove* an XR interaction is accessible via eye-tracking, switch scanning, or voice alone, addressing growing compliance requirements (EU Accessibility Act 2025, US Section 508); (5) a formal foundation upon which future XR interaction standards can be built.

## Technical Difficulty

The system demands simultaneous advances across compiler construction, computational geometry, and formal verification. Each subsystem below addresses a hard subproblem; their integration compounds the difficulty.

**Spatial type system design.** The type system must embed geometric reasoning: spatial subtyping via containment is undecidable for arbitrary regions but decidable for convex polytopes (reducible to linear programming). The language restricts to a decidable fragment—convex primitives, bounded CSG depth, Allen's 13 interval relations for temporal types—while remaining expressive enough for real interactions. The product lattice of spatial containment and temporal interval relations has non-trivial join/meet operations that must be computed at type-checking speed.

**Event Calculus compilation.** The compiler reverses the standard direction: prior work translates automata→EC for reasoning; Choreo compiles EC→automata for execution. The Discrete Event Calculus extended with a spatial oracle (continuous spatial predicates evaluated against an R-tree, with transitions fed as discrete events) must be faithfully lowered into guarded finite automata via a Thompson-style construction extended with spatial-temporal transition guards. Automata composition for multi-pattern choreographies requires on-the-fly product construction with symbolic state representation to contain the exponential blowup.

**Spatial CEGAR for verification.** The abstract state space is exponential (§Mathematics, M2–M3), so naive exploration is intractable. The verifier implements counterexample-guided abstraction refinement where the abstraction is *geometric*: merging nearby spatial regions, then splitting on spurious counterexamples that violate geometric realizability. This is fundamentally different from standard predicate abstraction—refinement requires solving spatial constraint satisfaction problems, not adding Boolean predicates.

**Incremental compilation with geometric dependencies.** Changing a spatial region definition can invalidate type-checking, EC evaluation, R-tree layout, and compiled automata simultaneously. The dependency graph is denser than in conventional compilers because geometric relationships create non-local invalidation. A Salsa-inspired demand-driven recomputation framework must track these spatial dependencies at expression granularity.

## New Mathematics Required

Five load-bearing mathematical results directly enable the artifact. Each is graded by novelty: **A** = would surprise experts in ≥1 established field; **B** = non-trivial new result combining known techniques in a novel way; **C** = routine application of known theory (still necessary, but not publishable as standalone math).

### M1: Spatial Event Calculus Formalization (Grade: C+)

**What it is.** A conservative extension of Mueller's Discrete Event Calculus (DEC) with a *spatial oracle*: a function σ: T → Scene mapping discrete timepoints to spatial configurations, from which *derived spatial fluents* (Proximity, Inside, GazeAt) are computed. Transitions of these fluents generate synthetic events fed into the standard EC axiom machinery. The central theorem is **sampling soundness**: for Lipschitz-continuous spatial trajectories with constant L and sampling resolution Δt < ε/(L·√3) for threshold ε, every spatial predicate transition in the continuous trace is detected within one tick.

**Why the artifact needs it.** The entire compiler rests on this formalization. Without a sound bridge between continuous 3D spatial state and discrete event processing, the compiler's correctness guarantees are meaningless—compiled automata could miss spatial transitions or generate spurious events. The conservative extension property ensures that restricting to inertial fluents recovers exactly DEC, so all classical EC results carry over.

**What proving it requires.** The sampling soundness theorem follows from the intermediate value theorem applied under the Lipschitz condition—the proof is elementary real analysis. The conservative extension is immediate by construction. The contribution is the clean formalization, not the proof technique. Estimated effort: 2–4 weeks.

### M2: Geometric Consistency Pruning for State-Space Reduction (Grade: B−)

**What it is.** The abstract state space for verification is S = Q₁ × ··· × Qₖ × 2^P, where P is the set of spatial predicates. Not all elements of 2^P are *geometrically realizable*: monotonicity (Proximity(a,b,r₁) → Proximity(a,b,r₂) for r₁ ≤ r₂), the triangle inequality (Proximity(a,b,r₁) ∧ Proximity(b,c,r₂) → Proximity(a,c,r₁+r₂)), and containment consistency (Inside(a,V₁) ∧ V₁ ⊆ V₂ → Inside(a,V₂)) jointly constrain the feasible predicate valuations to a set C ⊆ 2^P. **Theorem**: the reachable state space is contained in Q₁ × ··· × Qₖ × C, and |C| can be exponentially smaller than 2^|P|. For m proximity thresholds over n entities, monotonicity alone reduces 2^(n²·m) to O(m^(n²)).

**Why the artifact needs it.** Without geometric pruning, the verifier hits state-space explosion on any scene with more than ~10 interaction zones. The pruning makes the difference between a toy and a practical tool. The characterization of C is integrated directly into the BDD variable ordering and the CEGAR abstraction-refinement loop, guiding both symbolic and explicit-state exploration away from geometrically impossible configurations.

**What proving it requires.** Characterizing C precisely for the full predicate vocabulary (proximity, containment, gaze-cone intersection), proving tight cardinality bounds, and demonstrating an efficient membership test (likely via constraint propagation over a metric constraint network). The connection to clock-zone techniques in timed automata (UPPAAL's zone abstraction) provides proof structure, but the geometric instantiation—operating over ℝ³ metric constraints rather than clock differences—has not been studied. Estimated effort: 2–4 months. Risk: the bounds may not be tight enough to yield practically significant pruning for real XR scenes.

### M3: Spatial Tractability Theorem (Grade: B+conditional)

**What it is.** The verification problem for spatial-temporal event automata is PSPACE-complete in the worst case (standard product-automaton reachability). The **bet** is that XR scenes satisfy a natural structural property—*bounded spatial independence number*—that yields polynomial-time verification. Specifically: define the *spatial interference graph* G where nodes are interaction zones and edges connect zones whose spatial predicates can jointly influence a transition. If G has bounded treewidth w (plausible for XR scenes where most zones are spatially isolated: a hand menu doesn't interfere with a distant wall panel), then compositional verification along a tree decomposition of G reduces the state-space exploration from exponential in k (number of patterns) to polynomial in k for fixed w. **Theorem** (conjectured): for spatial-temporal event automata whose interference graph has treewidth ≤ w, reachability is decidable in time O(k · q^(w+1) · |C|) where q is the maximum per-pattern state count.

**Why the artifact needs it.** This theorem would transform the verifier from a tool that works on toy scenes (≤15 zones) to one that scales to production XR applications (50–200 zones) by exploiting the spatial locality that real scenes exhibit. The treewidth-based compositional technique is known from message-passing systems (Berwanger et al., Obdržálek); the novelty is the *domain identification*—that XR spatial structure yields bounded treewidth—and the adaptation of separator handling to geometric predicates. If the treewidth bound is empirically validated and the compositional algorithm correctly handles spatial separators, the result earns B+; promotion to A− would require a deeper structural insight beyond domain transfer.

**What proving it requires.** (1) Formally defining the spatial interference graph and proving that real XR scenes have bounded treewidth (empirical validation on the benchmark suite). (2) Constructing a tree-decomposition-based compositional verification algorithm that correctly handles the shared spatial predicates at tree-decomposition separators. (3) Proving the complexity bound. This is 6–12 months of focused theoretical work and carries genuine risk: the treewidth of real XR interference graphs may not be bounded, or the separator handling may not admit the conjectured complexity. If the theorem does not hold, the artifact remains viable with the B−grade geometric pruning (M2), but the best-paper argument weakens.

### M4: Spatial Type System Soundness (Grade: B)

**What it is.** The type system must guarantee that well-typed programs compile to *spatially realizable* automata: every guard in the compiled automaton is satisfiable by some geometric configuration consistent with the declared scene constraints. **Theorem**: if a Choreo program passes the spatial type checker, then for every transition guard g in the compiled automaton, there exists a scene configuration σ such that g(σ) = true. Contrapositively: spatially unrealizable interaction patterns (e.g., requiring an entity to be simultaneously inside two disjoint regions) are rejected at compile time.

**Why the artifact needs it.** Without this theorem, the compiler might produce automata containing dead transitions—guards that can never fire because the required spatial configuration is geometrically impossible. These dead transitions pollute verification (false deadlocks), waste runtime evaluation, and mislead developers. Soundness ensures that compilation preserves spatial intent.

**What proving it requires.** For the convex-polytope fragment, soundness reduces to linear programming feasibility (polynomial time). The interesting question is the *decidability boundary*: for which spatial fragments beyond convex polytopes does the soundness check remain decidable? Bounded-depth CSG (union/intersection/difference of convex primitives) is NP-complete in general but admits practical SAT encodings. Proving that the type checker is sound for the chosen fragment, with a tight complexity characterization, is a genuine PL+computational-geometry contribution. Estimated effort: 3–6 months.

### M5: End-to-End Compiler Correctness (Grade: B−)

**What it is.** The compiler translates DSL specifications to executable automata. Correctness means behavioral equivalence: for all specifications S and all traces τ, the compiled automaton accepts τ if and only if the specification's denotational semantics accepts τ. The proof proceeds by establishing simulation relations across each compilation phase (parsing → spatial type elaboration → EC intermediate representation → automaton construction → optimization). The critical composition theorem threads the Lipschitz sampling bound (M1) through the EC-to-automaton bisimulation: **Theorem (end-to-end)**: for any specification S, scene trace σ with Lipschitz constant L, and sampling rate Δt satisfying the M1 bound, the compiled automaton's behavior on the sampled trace agrees with the specification's continuous-trace semantics, up to timing error Δt.

**Why the artifact needs it.** The verification guarantees are only as trustworthy as the compiler. If compilation introduces or drops behaviors, then a "verified deadlock-free" choreography may deadlock in practice, or a "verified reachable" state may be unreachable. The end-to-end theorem is the correctness chain linking the developer's DSL specification to the verified automaton execution. Note that M5 composes M4 (spatial type soundness): the end-to-end proof relies on M4's guarantee that compiled transition guards are spatially realizable, ensuring the simulation relation maps only to geometrically feasible automaton states.

**What proving it requires.** Each phase requires a simulation relation proof. The individual steps are standard (parser generator correctness, desugaring by case analysis, bisimulation between EC fluent valuations and automaton states, R-tree query correctness from Guttman 1984). The composition is the non-trivial part: the sampling bound from M1 introduces a quantitative error parameter Δt that must be tracked through every subsequent phase, ensuring it does not amplify. The M4→M5 dependency is critical: the end-to-end proof relies on M4's guarantee that compiled transition guards are spatially realizable, ensuring the simulation relation maps only to geometrically feasible automaton states. This is not a new proof *technique* (CompCert established the methodology), but it is a new proof *instance* that requires care. Mechanization in Lean or Coq would elevate this from a routine proof to a notable verification artifact. Estimated effort: 3–5 months for pen-and-paper; 6–12 months for mechanization.

## Best Paper Argument

No language exists for XR interaction. The field has invested enormous effort in rendering (Vulkan, WebGPU, Metal), content interchange (glTF, USD, OpenUSD), and tracking abstraction (OpenXR)—but interaction choreography, the layer that determines what users *do* in a mixed-reality scene, remains trapped in imperative engine callbacks. Every new framework reinvents hand menus, gaze-dwell buttons, and grab mechanics from scratch. Choreo would be the first system to give interaction choreographies a formal syntax, static semantics, and a compilation target that is both executable and verifiable. This is a "new programming model" paper in the tradition of Cg (GPU shading → SIGGRAPH best paper), Halide (image processing pipelines → PLDI best paper), and TVM (ML compilation → OSDI best paper)—each of which won by revealing that an entire domain had been programming at the wrong abstraction level.

The formal-methods × spatial-computing intersection is largely unoccupied. The prior art audit found no existing tool that combines a declarative spatial-temporal language with formal verification of interaction properties. The closest systems differ categorically: **iv4XR** uses agent-based exploration with runtime assertions — empirical discovery, not exhaustive verification. **UPPAAL** verifies timed automata but has no spatial predicates, no XR-domain integration, and requires manual model construction (emerging spatial extensions remain research prototypes without XR-domain tooling). **SpatiaLang** provides a spatial interaction DSL compiled to engine code but lacks temporal reasoning, formal verification, headless execution, and compilation to verifiable automata. **Scenic** (PLDI 2019) compiles spatial-temporal scenario specifications with formal verification for autonomous driving, but targets scene *generation* rather than interactive multi-party choreography *verification*. **Multiparty Session Types** (MPST / Scribble) formalize choreographic communication protocols but are message-based (channel communication), not spatially-grounded (geometric predicate evaluation) — the type-theoretic machinery differs fundamentally. **Complex Event Processing** systems (Flink, GeoT-Rex) perform spatial-temporal pattern matching but lack formal verification, bidirectional interaction semantics, and compile-time guarantees.

From MPST: Choreo choreographies are spatially-grounded (geometric predicate evaluation over R-tree indices), not message-based (channel communication). The evaluation narrative seals the argument: extract interaction protocols from open-source Microsoft MRTK and Meta Interaction SDK samples, compile them with Choreo, run the verifier, and report bugs found—deadlocks and unreachable states that shipped in production toolkits used by millions of XR developers. ## Evaluation Plan

All evaluation is fully automated. No headsets, no human participants, no manual annotation.

**Bug-Finding on Real XR Projects.** Extract interaction protocols from MRTK canonical interaction components across ≥50 open-source XR projects: Microsoft MRTK sample scenes, Meta Interaction SDK examples, A-Frame/WebXR community projects, and Unity Asset Store XR samples. Extraction is scoped to ~15 supported MRTK component types (Interactable, NearInteractionGrabbable, ManipulationHandler, SolverHandler, BoundsControl, EyeTrackingTarget, and others with well-defined YAML schemas and documented state machines). Compile extracted protocols to Choreo automata and run the verifier. Report: (a) total deadlock states found, (b) unreachable interaction states, (c) race conditions (nondeterministic transitions on simultaneous spatial events), (d) temporal deadline violations. Cross-reference against each project's GitHub issue tracker to identify which flagged anomalies correspond to *known* bugs that shipped in production. Target: ≥5 interaction protocol anomalies found across open-source XR projects, with ≥2 corroborated by existing issue tracker reports. Note: the verifier operates on an abstraction of interaction logic that omits physics, animation, and rendering — flagged anomalies are interaction *protocol* defects (state machine deadlocks, unreachable states), not physics or rendering bugs.

**Compilation Throughput.** Measure patterns compiled per second, compiled automata state count vs. DSL specification complexity (number of interaction patterns, spatial zones, temporal constraints). Benchmark on a suite of 200+ interaction patterns parameterically scaled from trivial (single zone, single pattern) to complex (50+ zones, 20+ interacting patterns). Report compilation time breakdown by phase (parsing, type checking, EC lowering, automata construction, optimization).

**Verification Scalability.** Measure verification wall-clock time vs. abstract state-space size. Sweep parameters: number of interaction zones (5–100), number of concurrent patterns (2–30), number of spatial predicate templates. Compare: (a) naive explicit-state BFS, (b) BDD-based symbolic exploration without geometric pruning, (c) BDD-based exploration *with* geometric consistency pruning (M2), (d) spatial CEGAR, (e) compositional verification along the spatial interference graph (M3, if the tractability theorem holds). Report the pruning ratio |C|/|2^P| for each benchmark scene to quantify the effect of geometric structure on verification tractability. **Honest scalability ceiling**: without compositional verification (M3), the practical limit is ~15–20 interaction zones with verification completing in seconds to low minutes. Most MRTK sample scenes fall within this range (median ~8–12 zones). With compositional verification (M3, if the treewidth bound is validated), the practical limit extends to 50–100+ zones. Report the observed ceiling and the treewidth distribution of real XR interaction graphs.

**Runtime Trace Fidelity.** Compile DSL specifications to automata, execute headlessly against identical scenario scripts using the standalone Rust runtime, and verify determinism: repeated executions of the same scenario produce identical event traces. Additionally, implement a minimal platform adapter shim (~3K LoC) for one additional backend (Unity C# or WebXR JavaScript) to demonstrate cross-platform portability on a representative subset of 20 benchmarks. Define equivalence as identical event sequences with timestamps agreeing within a bounded skew ε. Report the maximum observed skew and the fraction of benchmarks achieving exact (ε=0) trace equivalence.

**Comparison Baselines.** (a) *iv4XR*: configure iv4XR's agent-based explorer on the same extracted interaction graphs; compare state-space coverage and bugs found. (b) *Manual UPPAAL encoding*: manually encode 10 representative interaction protocols as UPPAAL timed automata (no spatial predicates—the best UPPAAL can do) and compare verification time, expressiveness, and bugs found. (c) *Random simulation*: execute 10,000 random scenario traces per benchmark scene and compare the set of bugs found against the verifier's exhaustive results, quantifying the coverage gap between testing and verification.

## Laptop CPU Feasibility

The entire Choreo pipeline—from DSL source to verified automata—runs on a laptop CPU without GPU acceleration. This is not a limitation but a design choice that follows from the problem structure: the system processes *interaction logic*, not *rendered pixels*.

**Parsing and type checking** are string and graph operations: lexing, recursive descent, constraint propagation over a spatial subtyping lattice. These are CPU-native computations that complete in milliseconds for any realistic DSL specification.

**Compilation** (EC lowering, automaton construction, product composition, optimization) is graph construction and transformation. Thompson-style automaton construction is linear in specification size. Product composition is the most expensive phase, but on-the-fly construction with lazy state enumeration avoids materializing the full product; spatial independence partitioning reduces the effective dimensionality.

**R-tree construction and queries** are CPU-optimized spatial indexing. R*-trees with STR bulk-loading achieve cache-friendly memory layouts. For typical XR scenes (n ≤ 10⁴ entities), spatial queries complete in microseconds. The temporal parameterization adds a dimension but does not change the algorithmic character.

**Verification** (BDD-based symbolic model checking, SAT-based bounded model checking, CEGAR) is entirely CPU computation. BDD packages (CUDD, BuDDy) and SAT solvers (MiniSat, CaDiCaL) are mature, highly optimized CPU libraries. Geometric consistency pruning reduces the BDD variable count, and the spatial tractability theorem (M3) bounds the exploration to polynomial time for scenes with bounded spatial independence—precisely the regime where laptop CPUs are sufficient. Verification of a 20-zone scene with geometric pruning is estimated to complete in seconds to low minutes.

**Headless simulation** moves bounding volumes through space, detects collisions via GJK/EPA (CPU algorithms), and dispatches events based on spatial predicate transitions. No rasterization, no shading, no frame buffer. Simulation throughput of >10,000 frames/second on a single CPU core is achievable because the bottleneck is R-tree query time (microseconds) rather than rendering (milliseconds). A 1,000-scenario sweep of a moderately complex scene completes overnight on a laptop.

The system's CPU-only architecture is also a *practical advantage*: it enables integration into existing CI/CD pipelines (GitHub Actions, Jenkins, GitLab CI) running on standard cloud VMs without GPU provisioning—a critical requirement for the "headless XR testing" value proposition.
