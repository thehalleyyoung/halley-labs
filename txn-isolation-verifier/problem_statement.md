# IsoSpec: Verified Cross-Engine Transaction Isolation Analysis via Implementation Refinement

## 1. Problem Description

SQL database engines claim to implement the standard isolation levels defined in ANSI SQL-92 and refined by Berenson et al. (SIGMOD'95) and Adya's dependency serialization graphs (PhD thesis, MIT, 1999): READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, and SERIALIZABLE. In practice, every major production engine deviates from these specifications in subtle, dangerous, and well-documented ways. PostgreSQL 16.x implements SERIALIZABLE via Serializable Snapshot Isolation (SSI, Cahill et al., SIGMOD'08), which permits certain read-only anomalies that classical two-phase locking does not. MySQL 8.0 InnoDB uses next-key and gap locking whose behavior depends on the physical index structure—meaning the same logical transaction can observe different anomalies depending on whether an index exists. SQL Server 2022 offers both pessimistic (lock-based) and optimistic (snapshot-based) concurrency control as entirely separate code paths at the same nominal isolation level. Oracle 23c has never implemented true REPEATABLE READ, silently mapping it to its proprietary Multi-Version Read Consistency (MVRC). SQLite 3.x WAL mode provides snapshot isolation semantics that differ from every other engine's interpretation of SERIALIZABLE. These are not edge cases: they are the operational reality of every production SQL deployment, and they cause real bugs during database migrations, cloud transitions, and multi-engine architectures.

The state of the art in static analysis of transaction isolation anomalies—principally the CLOTHO tool (Rahmani et al., OOPSLA'19), Nagar and Jagannathan's framework (CONCUR'18), and related work on the Sieve tool (PLDI'21)—operates at the level of *abstract consistency models* for distributed data stores such as Cassandra with causal consistency or eventual consistency. These tools encode isolation specifications (e.g., "causal consistency" or "read atomicity") as SMT constraints and check whether a given transactional program can exhibit anomalies under that abstract model. This is pioneering work, and IsoSpec adopts CLOTHO's core architectural approach—parse transactional programs, encode isolation constraints, solve via SMT, extract counterexamples—while extending it in a fundamentally new direction. CLOTHO and related tools do not model the *implementation-level* behavior of any specific database engine. When CLOTHO checks serializability, it checks against Adya's DSG-based definition—not against PostgreSQL 16.x's SSI implementation, which is a strict subset of serializability with specific, documented gaps. No tool in the literature answers the question practitioners actually ask: "Is my transaction safe on *this* database engine at *this* isolation level, and will it break if I migrate to *that* engine?"

IsoSpec addresses this gap with three contributions. First, we provide formal operational semantics for the concurrency control mechanisms of five production SQL engines at specific, pinned versions: PostgreSQL 16.x (SSI with SIREAD locks and rw-dependency tracking), MySQL 8.0 InnoDB (next-key locking, gap locking, index-dependent phantom prevention), SQL Server 2022 (both lock-based and optimistic snapshot modes), Oracle 23c (MVRC with SCN-based versioning, modeled as approximate due to proprietary internals), and SQLite 3.x (WAL-mode snapshot isolation). These models capture the actual behavior of each engine's concurrency control *at the specified version*, validated against empirical results from the Hermitage project (Kleppmann, 2014), engine-specific regression test suites, and a formal adequacy criterion requiring behavioral agreement on bounded workloads. For optimizer-dependent behaviors (notably MySQL InnoDB gap locking, where lock acquisition depends on index selection), models adopt sound over-approximation: all possible index choices are considered, ensuring no false negatives at the cost of potential false positives. Second, we define a formal refinement relation connecting SQL standard isolation specifications (formalized via Adya's DSG framework) to these engine-specific operational models, with mechanized soundness proofs in Lean 4 for the PostgreSQL SSI model. Third, we implement a practical SMT-based verification tool that performs three analyses on user-provided SQL transaction programs: (a) single-engine anomaly detection, reporting which isolation anomalies are reachable under a specific engine's implementation; (b) cross-engine portability analysis, determining whether a transaction workload safe on engine E₁ remains safe when migrated to engine E₂; and (c) mixed-isolation optimization, computing the minimal per-transaction isolation levels required for safety on a given engine.

The tool's output is not merely a yes/no verdict. For each detected anomaly or portability violation, IsoSpec synthesizes a *minimal, runnable SQL test script* targeting the specific engine—complete with dialect-specific syntax, connection setup, and timing annotations—that demonstrates the anomaly on an actual database instance. This closes the loop between static analysis and dynamic validation: practitioners can execute the witness script against their production engine version to confirm the finding, and incorporate it into CI/CD pipelines as regression tests. This systematic, completeness-guaranteed, regression-testable approach to isolation verification addresses a gap that informal methods—including LLM-assisted analysis, which can enumerate common differences but cannot guarantee completeness, generate reproducible witnesses, or integrate into automated pipelines—cannot fill.

## 2. Value Proposition

Database migration is one of the leading causes of production outages and data corruption incidents in enterprise systems. When organizations migrate from Oracle to PostgreSQL (a transition driven by licensing costs and cloud-native strategies), from SQL Server to MySQL (common in cloud consolidation), or between any pair of production engines, transaction isolation behavior changes silently. An application tested and validated on Oracle's MVRC may silently produce inconsistent reads when deployed on PostgreSQL's default READ COMMITTED, which has subtly different visibility rules for concurrent updates. These bugs manifest as silent data corruption—balances that don't add up, inventory counts that go negative, audit trails with gaps—often weeks or months after migration, when root cause analysis is nearly impossible.

The cloud database migration market exceeds $10 billion annually, with AWS Database Migration Service, Azure Database Migration Service, and Google Cloud Database Migration Service as major products. These services handle schema translation and data transfer but perform *zero* verification of transaction isolation semantics. The AWS Schema Conversion Tool will translate Oracle PL/SQL to PostgreSQL PL/pgSQL, but it cannot tell you that your transaction logic depends on Oracle's specific MVRC behavior and will exhibit write skew anomalies on PostgreSQL at REPEATABLE READ. IsoSpec fills this gap with formal verification that is fully automated and produces actionable output.

Currently, developers and database administrators rely on a patchwork of documentation, blog posts (often outdated or incorrect), and trial-and-error testing to assess isolation behavior during migrations. The Hermitage project (Kleppmann, 2014) made a landmark empirical contribution by systematically testing isolation behaviors across six database engines, documenting dozens of divergences from the SQL standard. But Hermitage is a fixed set of hand-written test cases—it cannot analyze arbitrary user transactions, it cannot formally verify safety, and it provides no guarantee of completeness. Jepsen (Kingsbury) performs black-box testing of distributed databases but focuses on network partition behavior rather than isolation-level semantics of single-node SQL engines. Elle (Kingsbury & Alvaro, VLDB'20) checks observed histories against abstract isolation definitions but does not model engine-specific implementation mechanisms. IsoSpec is the first tool that combines formal engine models, static verification, and executable witness generation into a single workflow.

Beyond verification of user-provided transactions, IsoSpec's formal engine models serve as *discovery instruments*: by systematically exploring the behavioral space of each engine model and cross-validating against real engine instances, IsoSpec can identify engine-specific isolation behaviors not previously documented—including divergences beyond those cataloged by the Hermitage project. The evaluation includes a dedicated discovery component targeting stored procedure interactions, trigger-induced anomalies, multi-table foreign key transactions, and index-dependent behaviors, with all discovered divergences confirmed by dynamic execution on real engine instances.

## 3. Technical Difficulty

The implementation comprises approximately 150K lines of code, of which roughly 55K are genuinely novel engine-specific and theory code, ~15K are novel applications of known techniques to the isolation domain, and ~80K are parser infrastructure, test harnesses, benchmarks, and CLI tooling. The hard subproblems are:

**Formal Engine Models (~25K LoC novel).** Modeling the operational semantics of five production database engines' concurrency control mechanisms is the core intellectual challenge. Each model is pinned to a specific engine version and validated against a formal adequacy criterion requiring behavioral agreement with the real engine on all bounded workloads up to a specified size. PostgreSQL 16.x's SSI implementation—based on Cahill, Röhm, and Fekete (SIGMOD'08, TODS'09)—involves SIREAD locks, rw-dependency graph maintenance, a "dangerous structure" detection algorithm, and a read-only optimization that permits certain schedules other SSI implementations would abort. MySQL 8.0 InnoDB's gap locking is *index-structure-dependent*: the set of phantom-preventing locks acquired by a range scan depends on which index the query optimizer selects, meaning the same SQL statement can have different isolation behavior depending on the physical schema. IsoSpec models this via sound over-approximation, considering all possible index selections to ensure no false negatives. SQL Server 2022 provides two fundamentally different concurrency control mechanisms (pessimistic locking vs. optimistic snapshot isolation via row versioning in tempdb) selected by a database-level configuration flag, requiring two complete models at the same nominal isolation level. Oracle 23c's MVRC uses System Change Numbers (SCNs) and undo segments; this model is labeled *approximate* because Oracle's concurrency control internals are proprietary and the model is based on observed behavior and limited documentation rather than source code analysis. SQLite 3.x's WAL mode implements a form of snapshot isolation with database-level write serialization that is simpler than other engines but has unique implications for concurrent readers.

**SQL Parser with Dialect Extensions (~6K LoC novel).** Building on the `sqlparser-rs` crate (which handles standard SQL and most dialect variations including PostgreSQL, MySQL, SQL Server, and SQLite), this subsystem extends the parser with isolation-relevant features: engine-specific locking syntax (PostgreSQL's `FOR UPDATE SKIP LOCKED`, MySQL's `LOCK IN SHARE MODE`, SQL Server's lock hints), and the translation layer to IsoSpec's internal IR. Version 1 handles pure SQL transactions; stored procedures, triggers, and procedural extensions (PL/pgSQL, T-SQL, PL/SQL) are deferred to follow-on work due to the complexity of analyzing four distinct procedural languages.

**SMT Encoding of Engine Semantics (~8K LoC novel).** Using the Rust `z3` crate bindings, this subsystem encodes operational semantics—lock tables with compatibility matrices, version stores with visibility rules, SIREAD lock sets with dependency graph analysis—as quantifier-free SMT constraints over LIA+UF+Arrays. This adopts the bounded model checking approach established by CLOTHO and related tools, extending it with engine-specific constraint content. The key engineering challenges are: (a) encoding gap-lock ranges as symbolic intervals with sound over-approximation of index choice, (b) encoding PostgreSQL SSI's read-only optimization (which requires reasoning about the *absence* of certain dependency patterns), and (c) encoding Oracle's SCN-based visibility with undo segment bounds. The analysis is *sound* (every reported anomaly is a real anomaly under the engine model) but *bounded-complete* (completeness holds only up to the specified bound on transaction instances and loop iterations).

**Refinement Theory Implementation (~5K LoC novel).** The refinement relation ⊑ between specification-level isolation (Adya DSG) and engine-level operational semantics must be computed for each of 5 engines × 4 isolation levels = 20 pairs, with proofs that the engine model is a sound refinement of the specification. This applies established refinement theory from process algebra to the new domain of SQL isolation verification, requiring both an algorithmic decision procedure and mechanized proofs of its correctness for the PostgreSQL SSI model.

**Cross-Engine Portability Analyzer (~6K LoC novel).** Differential analysis across all 10 unordered engine pairs (C(5,2) = 10) requires computing the *difference* in reachable anomaly sets between two engine models on a given transaction workload—a problem that reduces to differential SMT satisfiability with shared symbolic transaction structures but distinct engine constraint sets.

**Witness Synthesis (~4K LoC novel).** Generating engine-specific, runnable SQL test scripts from SMT counterexamples requires mapping symbolic schedules back to concrete SQL with dialect-correct syntax, appropriate `SET TRANSACTION ISOLATION LEVEL` statements, connection management, and timing annotations that reliably reproduce the interleaving. Witness irredundancy (no proper sub-schedule exhibits the same anomaly) is ensured via iterative MUS (Minimal Unsatisfiable Subset) extraction.

**Lean 4 Proofs (~9K LoC).** Mechanized soundness proofs for: (a) the refinement relation between Adya DSG specifications and the PostgreSQL 16.x SSI engine model (the flagship engine, fully mechanized), (b) the correctness of the SMT encoding relative to the PostgreSQL operational semantics, and (c) the compositionality theorem enabling incremental portability analysis. The remaining four engine models have paper proofs with key lemmas mechanized; full mechanization of all five engines is deferred to follow-on work. The PostgreSQL SSI model alone requires formalizing Cahill et al.'s dangerous structure theorem.

**Transaction IR (~7K LoC novel).** An isolation-aware intermediate representation that normalizes five SQL dialects into a common form suitable for analysis while preserving isolation-relevant information (lock acquisition points, version read points, predicate ranges).

**Mixed-Isolation Recommender (~3K LoC novel).** Per-transaction isolation level computation formalized as a weighted-cost optimization modulo theories (MaxSMT), where each isolation level is assigned a cost reflecting its performance impact and the solver minimizes total cost subject to the engine-specific safety constraints.

**Engine Model Validation Oracle (~3K LoC novel).** Automated cross-validation infrastructure that tests each formal engine model against the real engine's behavior at the pinned version. For each engine, generates parameterized test schedules from the LTS, executes them on Dockerized engine instances using controlled concurrency (thread barriers and advisory locks to force specific interleavings), and checks that the engine's observed behavior matches the model's predictions. Interleaving-forcing is inherently approximate due to non-deterministic engine-internal scheduling; the validation oracle may have false negatives but is designed to catch systematic model errors. Covers regression testing when engine versions are updated.

**Test Suite, Benchmarks, and CLI (~80K LoC infrastructure).** Including the Hermitage benchmark suite, TPC-C/TPC-E transaction workloads, anomaly regression tests, end-to-end integration tests, real-world migration case study encodings, CLI with reporting infrastructure, and CI/CD integration scaffolding.

## 4. New Mathematics Required

IsoSpec's theoretical contributions are organized in three tiers: one flagship theorem, one framework contribution comprising two interdependent results, and supporting technical results that enable the implementation.

### Tier 1: Flagship Theorem

**M5: Predicate-Level Conflict Theory for Range Operations.** Extend Adya's item-level dependency serialization graph theory to handle SQL WHERE clauses, range scans, INSERT/DELETE operations that affect predicate-defined sets, and phantom-susceptible operations. Define *predicate conflicts* as symbolic overlap conditions between WHERE clause predicates, and encode these as SMT constraints. The decidable fragment handled is the conjunctive inequality fragment over numeric and string columns (covering the vast majority of practical WHERE clauses in OLTP workloads); arbitrary user-defined functions in predicates are conservatively over-approximated. Adya §3.4 sketches predicate-level dependencies but provides no symbolic encoding. Fekete et al. (TODS'05) analyze snapshot isolation with predicate reads but not in a multi-engine symbolic setting. This is the most technically novel contribution: the decidability boundary characterization and the multi-engine predicate conflict encoding are genuinely new results. Difficulty: **Hard**.

### Tier 2: Framework Contribution

**M1: Engine Isolation Operational Semantics.** For each of five engines at pinned versions, a formal operational semantics of the concurrency control mechanism expressed as a labeled transition system. States comprise engine-specific structures: lock tables with mode hierarchies (PostgreSQL 16.x), lock tables with gap/next-key annotations indexed by B-tree intervals with sound over-approximation of index choice (MySQL 8.0 InnoDB), version stores with SCN-based visibility predicates (Oracle 23c, approximate model), dependency graphs with SIREAD lock annotations (PostgreSQL SSI), and WAL-mode snapshot descriptors (SQLite 3.x). Transitions correspond to SQL operation execution, lock acquisition/release, version creation/garbage collection, and abort/commit. While Ports and Groves (ESOP'10) formalize snapshot isolation operationally and Cahill et al. (TODS'09) describe SSI algorithmically, no published paper provides implementation-faithful operational semantics capturing vendor-specific mechanisms such as InnoDB's index-dependent gap locking, Oracle's SCN-based undo visibility, or PostgreSQL SSI's read-only optimization. Model adequacy is defined via trace inclusion on bounded workloads: for all transaction programs with at most k transactions and n operations per transaction, the set of commit/abort outcomes admitted by the LTS must agree with outcomes observable on the real engine at the pinned version. This is a *formalization contribution* — the novelty lies in the artifacts produced (five formal engine models that have never existed), not in new mathematical technique. Difficulty: **Hard** (engineering and domain expertise).

**M2: Isolation Refinement Relation.** A formal refinement relation ⊑ between specification-level isolation definitions (Adya's DSG-based characterization: G0, G1a, G1b, G1c, G2-item, G2) and engine-level operational semantics. Engine E at isolation level I *refines* specification S iff every schedule permitted by E's operational semantics under configuration I satisfies the anomaly-freedom properties required by S. We provide a decision procedure for computing this relation and prove its correctness. This applies established refinement theory from process algebra (Milner, Abramsky) to the new domain of SQL isolation verification — the novelty is in the specific application and the decision procedure, not in the refinement concept itself. Cerone et al. (CONCUR'15), Biswas and Enea (CAV'19, PLDI'21), and Rahmani et al. (OOPSLA'19) all work at the specification level without connecting to implementation-level models. Difficulty: **Moderate** (known theory, novel application).

### Tier 3: Supporting Technical Results

**M3: Cross-Engine Portability Theorem.** A transaction set T is *portable* from engine E₁ at isolation level I₁ to engine E₂ at isolation level I₂ iff the set of anomalies reachable under E₂/I₂ restricted to T's joint data access footprint is a subset of those reachable under E₁/I₁. We prove that this portability checking problem is coNP-complete for bounded transactions. This is an expected complexity-theoretic result that extends Fekete et al.'s results and relates to Brutschy et al. (POPL'17). Difficulty: **Moderate**.

**M4: Symbolic Engine Constraint Encoding.** For each engine model, a polynomial-size encoding in QF_LIA+UF+Arrays that is equisatisfiable with the engine's operational semantics restricted to a given set of bounded transactions. This is hard engineering within the established bounded model checking framework, applying known SMT encoding techniques to novel engine-specific constraint content. Difficulty: **Hard** (engineering).

**M6: Mixed-Isolation Optimization.** Given N transactions on engine E with a safety specification, find the minimum-cost assignment of per-transaction isolation levels formalized as weighted partial MaxSMT. Engine-specific isolation levels form a DAG, not a total order—SQL Server's SNAPSHOT is incomparable with REPEATABLE READ in anomaly prevention. The formalization is novel for the isolation-level domain; the MaxSMT solving technique is established. Difficulty: **Moderate**.

**M7: Bounded Soundness and Completeness Characterization.** The standard bounded model checking guarantee applied to this domain: sound unconditionally, complete up to the bound. The specific result that k=3 suffices for all Adya anomaly classes G0–G2 follows from the structure of Adya's anomaly definitions (which involve 2–3 transaction interactions). Difficulty: **Minor** (expected result).

**M8: Compositionality Theorem.** If workload W is verified safe under engine E₁ at isolation level I, and E₂ at level I' refines E₁ at I when restricted to footprint(W), then W is safe under E₂ at I'. This is the standard monotonicity property of refinement restricted to a footprint, useful for incremental portability analysis. Difficulty: **Minor** (standard property of refinement).

## 5. Best Paper Argument

IsoSpec merits best-paper consideration on six grounds. First, it solves a **real and urgent problem**: database migrations are a leading cause of production outages, isolation divergences cause silent data corruption during migrations, and no existing tool addresses this formally. Second, the formal engine models serve as **discovery instruments**, systematically revealing engine-specific isolation behaviors beyond those documented by the Hermitage project — each newly discovered divergence, confirmed by dynamic execution, is an independent contribution to the database community's understanding of production engine behavior. Third, it **bridges theory and practice** in a way that the best PL/DB papers do: it connects Adya and Berenson's theoretical isolation framework to the messy reality of production engine implementations, via a clean refinement theory. Fourth, it provides the **first implementation-faithful formalization of engine-specific isolation semantics**: while Ports and Groves (ESOP'10) formalize snapshot isolation operationally and Cahill et al. (TODS'09) describe SSI algorithmically, no published paper provides operational semantics capturing vendor-specific mechanisms such as InnoDB's index-dependent gap locking or PostgreSQL SSI's read-only optimization. Fifth, it is **honestly positioned** relative to prior art: we adopt CLOTHO's architectural approach (Rahmani et al., OOPSLA'19) and extend it from abstract consistency models for distributed stores to engine-specific implementation models for production SQL databases, connected via a formal refinement bridge. Sixth, it **closes the verification loop**: generated SQL test scripts can be executed directly against production database instances, integrated into CI/CD pipelines, and used as regression tests during migration projects.

The predicate-level conflict theory (M5) represents a genuinely novel theoretical contribution: extending Adya's item-level DSG theory to handle range operations with a characterized decidability boundary in a multi-engine symbolic setting.

## 6. Evaluation Plan

All evaluation is fully automated with no human annotation required.

**Known Anomaly Benchmark.** We curate a benchmark of approximately 200 known isolation anomalies drawn from: the Hermitage project's empirical test suite (Kleppmann, 2014), Jepsen reports for SQL-compatible databases, PostgreSQL and MySQL bug tracker entries tagged with isolation/concurrency keywords, and anomaly examples from the academic literature (Berenson et al., Adya, Fekete et al., Cahill et al.). For each anomaly, we encode the transaction program and expected behavior. We measure detection recall (does IsoSpec flag the anomaly?) and precision (does IsoSpec avoid false positives on safe variants?).

**Novel Divergence Discovery.** We systematically explore the behavioral space of each engine model beyond Hermitage patterns, targeting: multi-table transactions with foreign key constraints, index-dependent behaviors in MySQL beyond Hermitage's coverage, and edge cases in PostgreSQL SSI's read-only optimization. For each model-predicted divergence not in the Hermitage catalog, we execute confirmation tests on real engine instances. We report the number of newly discovered divergences and their practical significance.

**Real-World Migration Case Study.** We identify 1–3 documented production migration failures from PostgreSQL mailing list archives, AWS DMS-related bug reports, and database migration guides. For each case, we encode the triggering transaction workload, run IsoSpec, and demonstrate that the tool produces a witness script reproducing the migration-induced bug. This evaluation component directly demonstrates practical value.

**Cross-Engine Portability Benchmark.** We extract transaction workloads from TPC-C (5 transaction types), TPC-E (10 transaction types), and OLTPBench (15 benchmarks) and analyze portability across all 10 unordered engine pairs at each isolation level. We report: number of portability violations found, anomaly types involved, and whether violations correspond to known engine behavioral differences.

**Ground Truth Validation (Closing the Loop).** For every anomaly or portability violation reported by IsoSpec, we automatically execute the generated SQL witness script on the target database engine (running in Docker containers with pinned versions) and verify that the anomaly manifests. We report the confirmation rate (expected: >95% for core models; lower for Oracle approximate model and for edge cases involving optimizer non-determinism). The validation oracle uses advisory locks and thread barriers for interleaving control; we acknowledge that fine-grained interleaving forcing is inherently approximate, and report the validation coverage honestly.

**Comparison with Prior Tools.** On anomalies detectable by CLOTHO (serializability violations under abstract models), we compare detection time and witness quality. On engine-specific anomalies (e.g., MySQL gap-lock-dependent phantoms, PostgreSQL SSI read-only optimization edge cases, Oracle MVRC-specific write skew), we demonstrate that CLOTHO, Elle, and Jepsen's checkers cannot detect them because they lack engine-specific models. We provide a feature matrix comparing IsoSpec against CLOTHO, Elle, Jepsen, dbcop (Biswas & Enea), and MonkeyDB.

**Scalability.** We measure end-to-end verification time as a function of: number of transactions (2–100), operations per transaction (1–50), number of tables and columns, and schema complexity (number of indexes). We report times for each engine model separately, since model complexity varies (MySQL InnoDB gap locking with over-approximated index choice generates more constraints than SQLite WAL). Expected regime: seconds for typical workloads (5–20 transactions), minutes for stress tests (50–100 transactions). We report Z3 timeout rates for pathological inputs.

**Soundness Validation (False Positive Rate).** For transactions verified as "safe" by IsoSpec, we perform exhaustive dynamic testing using randomized schedule exploration on real engine instances. We run each safe-verified workload under 10,000 randomized schedules and check for anomaly manifestation. Any manifestation would indicate a soundness bug in IsoSpec's engine model. Note: for MySQL gap-locking, sound over-approximation means IsoSpec may report anomalies that do not manifest under typical optimizer decisions; we measure and report this over-approximation rate separately.

## 7. Laptop CPU Feasibility

IsoSpec is designed for single-machine, CPU-only execution. The core SMT solving uses Z3 (de Moura & Bjørner), which is highly optimized for CPU execution and is the standard backend for virtually all SMT-based program analysis tools. The symbolic state space is bounded by the number of transactions and operations in the input program, not by data volume or database size. Real-world SQL transaction programs typically involve 2–20 concurrent transactions with 5–50 operations each; the resulting SMT formulas are well within the tractability envelope of modern SMT solvers on laptop hardware (seconds to minutes for typical workloads). The bounded completeness result (k=3 suffices for all Adya anomaly classes) ensures that completeness does not require exploring large transaction sets.

Engine-specific models add SMT constraints but also *prune* the search space relative to abstract models: lock compatibility matrices, version visibility rules, and engine-specific abort conditions eliminate large classes of interleavings that abstract models must consider. This means that engine-aware analysis is often *faster* than abstract analysis for the same transaction workload, because the engine model constrains the set of feasible schedules. MySQL gap-lock over-approximation (considering all possible index choices) adds disjunctive constraints, but for typical transaction sizes the overhead is manageable; we report Z3 timeout rates for stress-test inputs.

Lean 4 proof checking is CPU-only and completes in minutes for proof artifacts of this size. All evaluation—SQL parsing, SMT solving, witness generation, and dynamic validation against Dockerized database instances—runs on a single machine. The Docker-based ground truth validation requires running five database engines in containers sequentially (not simultaneously), which is standard on any modern laptop with 16GB RAM. PostgreSQL (~200MB), MySQL (~400MB), SQL Server (~1.5GB), Oracle XE (~2GB), and SQLite (native, no container) total ~4.1GB when run sequentially. No GPU computation is required. No human annotation is required. The entire pipeline from input SQL to verified output with executable witnesses is fully automated.
